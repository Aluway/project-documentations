# 04 — Releases

> Оглавление: [`README.md`](README.md).

Правила релизного цикла: semver, changelog, tagging, hotfix-процесс, откаты. Применяются к коду (не к документации — там версионирование per-глава, см. [`../_meta/frontmatter.md`](../_meta/frontmatter.md) раздел 2).

---

## 1. Semver

Версии кода — по [Semantic Versioning 2.0.0](https://semver.org/): `MAJOR.MINOR.PATCH`.

| Инкремент | Когда |
|---|---|
| **MAJOR** (`1.x.x` → `2.0.0`) | Breaking change публичного контракта: удалён/переименован API, изменено поведение, на которое опираются потребители. |
| **MINOR** (`1.2.x` → `1.3.0`) | Новая функциональность, обратно совместимая. |
| **PATCH** (`1.2.3` → `1.2.4`) | Исправление бага, обратно совместимое. Нет новой функциональности. |

### Правила

- **MUST** — каждый релиз имеет версию по semver. «Дата-версии» (`2026.04.17`) или календарные — **MUST NOT** для кода (кроме явного CalVer-решения в отдельном ADR).
- **MUST** — breaking change **MUST** приводить к MAJOR-bump. Это верно даже для проектов с одним потребителем — semver защищает от неявных допущений.
- **SHOULD** — breaking change **SHOULD** сопровождаться migration guide (отдельная секция в changelog или отдельный файл `docs/migrations/`).
- **MAY** — pre-1.0 (`0.x.x`) трактуется как «API нестабилен»: MINOR может содержать breaking changes. После первого прода **SHOULD** перейти на 1.0.0.

### Что считать «публичным контрактом»

- Для библиотеки: экспортируемые функции, классы, типы, константы.
- Для приложения: публичные HTTP/API endpoints, URL-схемы роутов, формат persisted-данных (localStorage, cookies, DB-schema), формат конфигурации.
- Для монорепо: контракт между пакетами.

Внутренние детали реализации (private-функции, конкретная структура slice'ов) — **MUST NOT** включаются в public contract и **MAY** меняться без semver-bump.

## 2. Changelog

### Формат

- **MUST** — файл `CHANGELOG.md` в корне репозитория.
- **MUST** — формат [Keep a Changelog](https://keepachangelog.com/): секции `Unreleased`, затем релизы в обратном хронологическом порядке.
- **MUST** — подсекции внутри релиза:
  - `Added` — новая функциональность.
  - `Changed` — изменения существующего поведения (не баги).
  - `Deprecated` — помеченное к удалению.
  - `Removed` — удалённая функциональность.
  - `Fixed` — исправленные баги.
  - `Security` — исправления уязвимостей.

Пустые подсекции **MUST NOT** оставляться.

### Пример

```markdown
# Changelog

All notable changes to this project are documented in this file.
Format: Keep a Changelog 1.1.0. Versioning: Semver 2.0.0.

## [Unreleased]

### Added
- Promo code redemption on checkout (#142).

## [1.4.0] — 2026-04-15

### Added
- OAuth login via Google (#128).

### Fixed
- Cart crash on empty basket during checkout (#135).

## [1.3.1] — 2026-03-30

### Fixed
- Date formatting in Safari below 17 (#121).

## [1.3.0] — 2026-03-22
...
```

### Когда обновлять

- **MUST** — каждый PR, меняющий поведение, **MUST** добавить запись в `## [Unreleased]`. Записи не в Unreleased-секции правятся только при релизе.
- **MAY** — тривиальные изменения (правка typo в docs, внутренние рефакторинги без изменения поведения) **MAY** пропускать changelog-entry.
- **MUST** — запись пишется на английском, в повелительно-результативной форме: «Add promo code redemption», не «Added the functionality of promo codes».

### Ссылки на issues / PRs

Каждая запись в changelog **SHOULD** иметь ссылку на PR или issue в скобках: `(#142)`. Это превращает changelog в навигацию по истории решений.

## 3. Tagging

### Формат тега

- **MUST** — префикс `v` + semver: `v1.2.3`.
- **MUST NOT** — теги без префикса (`1.2.3`) — конфликтуют с git-рефами в некоторых тулингах.
- **MUST NOT** — неформальные теги в релизной истории (`latest`, `stable` как отдельные теги — **MAY** только если они аннотированные и явно документированные).

### Процесс релиза

1. Убедиться, что `main` зелёный и все PR, которые должны войти в релиз, смёржены.
2. В отдельном PR «release: vX.Y.Z»:
   - Обновить версию в `package.json` (и аналогах в других пакетах монорепо).
   - Перенести `## [Unreleased]` → `## [X.Y.Z] — YYYY-MM-DD`.
   - Создать новую пустую `## [Unreleased]`.
3. Смёржить release-PR в `main` (squash-merge).
4. Создать аннотированный тег на merge-commit: `git tag -a v1.2.3 -m "Release 1.2.3"`.
5. `git push origin v1.2.3`.
6. Создать Release на хостинге (GitHub Release / GitLab Release) с копией секции changelog за этот релиз.

### Аннотированные vs lightweight теги

- **MUST** — аннотированные теги (`git tag -a`) для релизов. Содержат автора, дату, сообщение, подпись.
- **SHOULD NOT** — lightweight-теги для релизов; они не отличимы от веток в `git show-ref` и не содержат метаданных.

### Подпись тегов

- **SHOULD** — тег подписан GPG/SSH-ключом мейнтейнера (`git tag -s`). Это верифицирует авторство релиза.
- **MUST NOT** — обходить требование подписи через `-c tag.gpgsign=false` без согласия owner'а.

## 4. Release branches (опционально)

Некоторые проекты используют release-ветки для поддержки предыдущих мажорных версий (например, LTS).

- **MAY** — если проект требует параллельной поддержки нескольких версий.
- Именование: `release/vX.Y` (без `.Z` — внутри ветки идут PATCH-релизы).
- Исправления портируются из `main` в release-ветку через `git cherry-pick`.
- Новые фичи в release-ветки **MUST NOT** попадать — только fixes.

Template-репо по дефолту использует **trunk-based** модель: одна ветка `main`, теги на merge-коммитах. Release-ветки вводятся отдельным ADR в форкнутом проекте, если появляется потребность.

## 5. Hotfix-процесс

Критический баг в продакшене, требующий исправления быстрее стандартного PR-цикла.

### Процедура

1. Открыть ветку `hotfix/v<current>-<short-desc>` от тега текущего релиза (или от `main`, если `main` — текущий прод).
2. Написать **regression-тест**, воспроизводящий баг. Зелёный тест фиксирует начало работы.
3. Сделать минимальное исправление — только то, что необходимо для фикса. Никакого сопутствующего рефакторинга.
4. Open PR с заголовком `fix: <short-desc>` и префиксом `[hotfix]` в описании.
5. Ускоренный review (SLA: 2 часа в рабочее время, см. [`03-code-review.md`](03-code-review.md) раздел 2).
6. Merge → тег с PATCH-bump (`v1.2.3` → `v1.2.4`).
7. Deploy.
8. **MUST** — post-mortem issue с root cause, impact, preventive action. Открывается в течение 3 рабочих дней после инцидента.

### Что НЕ делать в hotfix

- **MUST NOT** — расширять scope за пределы минимального фикса.
- **MUST NOT** — скипать regression-тест «потому что некогда». Без теста баг вернётся.
- **MUST NOT** — skipping CI / admin-merge, даже при срочности. Красный CI — сигнал о большем, чем известный баг.

## 6. Откаты

Если релиз выявил проблему, несоразмерную быстрому hotfix'у:

### Откат через тег

- Ведёт к deployment предыдущего тега. Коммиты в `main` остаются, но в прод едет `v1.2.2` вместо `v1.2.3`.
- **MAY** как временная мера, пока готовится `v1.2.4`.

### Revert-коммит

- `git revert <sha>` создаёт новый коммит, отменяющий изменение. История сохраняется, semver-bump минимум PATCH.
- Revert-коммит **MUST** применяться вместо force-push / reset для отката в `main` — история сохраняется, клоны и PR не ломаются.

### Force-откат

- `git reset --hard <old-sha>` + force-push в `main` — **MUST NOT** без явного согласия всех активных контрибьюторов и уведомления о необходимости переклонировать. Ломает клоны и PR.

## 7. Запрещённые паттерны

- **Релиз без обновления CHANGELOG.md** — **MUST NOT**.
- **Тег, не соответствующий версии в `package.json`** — **MUST NOT**. Они обновляются атомарно.
- **Два релиза с одной версией** — **MUST NOT** (semver это запрещает — версия immutable).
- **Pre-release (`1.2.3-rc.1`) без явной пометки** — **MUST NOT** попадать в production deployment как стабильный.
- **Changelog-entry после факта** («задним числом при релизе») — **SHOULD NOT**. Entry пишется в том же PR, что и само изменение, в секции `Unreleased`.
- **Breaking change без MAJOR-bump** — **MUST NOT**. Даже если «никто этим не пользуется» — предположение опасно.
- **Релизный PR, содержащий функциональные изменения помимо bump'а версии и changelog** — **MUST NOT**. Release PR — чистая подготовка к тегу, не место для новых фич.

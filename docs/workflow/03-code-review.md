# 03 — Code Review

> Оглавление: [`README.md`](README.md).

Протокол ревью: кто ревьюит, SLA, чек-лист ревьюера, формы feedback, критерии approve, обращение с разногласиями.

---

## 1. Кто ревьюит

### Минимальное число ревьюеров

- **MUST** — хотя бы **один** approve от человека, не являющегося автором PR.
- **SHOULD** — **два** approve для PR, затрагивающих: security-границы (auth, permissions, secrets handling), публичные контракты API, миграции данных, релизный процесс, `docs/_meta/`, новые ADR со `status: accepted`.
- PR в solo-репо (один мейнтейнер) — **MAY** мёржиться без внешнего approve; в этом случае **MUST** быть пауза ≥ 1 часа между открытием и мёржем (self-review после перерыва).

### Выбор ревьюера

- **SHOULD** — ревьюер знаком с затронутой областью кода. Для FSD-слайсов — автор прошлых коммитов в этом слайсе (смотрите `git log --author`).
- **MUST** — если у категории есть владелец (`CODEOWNERS` или [`../_meta/governance.md`](../_meta/governance.md) раздел 1), владелец **MUST** быть включён в ревьюеры при изменении своей категории.
- **SHOULD NOT** — один и тот же человек ревьюит все PR одного автора подряд (эффект «rubber-stamp»). Ротируйте.

## 2. SLA

Временные ожидания для ревью. Рабочие часы — дни автора по его таймзоне.

| Действие | Target |
|---|---|
| **First response** — либо ревью, либо явное «не смогу, передай другому» | ≤ 1 рабочий день с момента request-review. |
| **Full review** — по всему PR | ≤ 2 рабочих дня. |
| **Re-review после правок автора** | ≤ 1 рабочий день. |
| **Urgent / hotfix** (помечен в заголовке `hotfix/`) | ≤ 2 часов в рабочее время. |

- Если ревьюер не может уложиться — **MUST** явно ответить (комментом или через delegate) и предложить другого. Тишина > 2 рабочих дней — **SHOULD NOT**.
- Автор PR **MAY** пинговать ревьюера после истечения SLA в канале согласования команды.

## 3. Чек-лист ревьюера

Перед нажатием «Approve» ревьюер **MUST** проверить:

### Корректность

- [ ] Код делает то, что заявлено в Summary / Why PR.
- [ ] Граничные случаи обработаны (пустой список, `null`/`undefined`, ошибки сети, race conditions).
- [ ] Нет очевидных багов: off-by-one, незакрытые ресурсы, утечки, вечные циклы.
- [ ] Типы корректны; `any` оправдан, если присутствует ([`../code-style/universal/01-typescript.md`](../code-style/universal/01-typescript.md)).

### Архитектура и стиль

- [ ] Изменения соответствуют [`../architecture/AGENTS.md`](../architecture/AGENTS.md) Hard Invariants: направление слоёв, Public API, pages-first.
- [ ] Изменения соответствуют [`../code-style/AGENTS.md`](../code-style/AGENTS.md) Hard Invariants и активным `variants/*`.
- [ ] Нет «умных» абстракций в ответ на одноразовую задачу (DRY ≠ преждевременное обобщение).

### Тесты

- [ ] Для новой функциональности — есть тесты.
- [ ] Для фикса бага — есть regression-тест, воспроизводящий баг до фикса.
- [ ] Тесты проверяют **поведение**, а не реализацию (см. `universal/08-testing-principles.md`).
- [ ] Моки и фикстуры не заглушают реальные ошибки (mock не должен возвращать данные, которые в проде невозможны).

### Безопасность

- [ ] Нет секретов, `.env`-значений, API-ключей в коде или комментариях.
- [ ] Внешний input валидируется на границе.
- [ ] Нет `dangerouslySetInnerHTML`, `eval`, `new Function(...)` без обоснования.

### Документация

- [ ] Публичные функции / компоненты / типы документированы (короткий JSDoc).
- [ ] Если изменение ломает публичный контракт — CHANGELOG / release-note обновлены ([`04-releases.md`](04-releases.md) раздел 2).
- [ ] Если PR вводит структурное решение — есть ADR в том же PR.

### Гигиена PR

- [ ] Заголовок и описание по [`02-pull-requests.md`](02-pull-requests.md) раздел 2–3.
- [ ] Коммиты атомарны ([`01-git.md`](01-git.md) раздел 3).
- [ ] CI зелёный.
- [ ] Нет конфликтов с `main`.

## 4. Формы feedback

Каждый комментарий ревьюера **MUST** иметь префикс, обозначающий силу требования. Это снимает двусмысленность: автор точно знает, что надо чинить, а что — опционально.

| Префикс | Сила | Пример |
|---|---|---|
| `[blocker]` | **MUST fix** перед мёржем. | `[blocker] This mutates the input array, which breaks the calling code in features/checkout.` |
| `[issue]` | **SHOULD fix** или явно обсудить. | `[issue] The retry loop will busy-loop on 429 — add backoff.` |
| `[suggestion]` | **MAY fix**. Улучшение, но не обязательно. | `[suggestion] Could be expressed with reduce; current version is fine.` |
| `[nit]` | Стилистика, опечатки. Автор **MAY** игнорировать. | `[nit] Extra whitespace.` |
| `[question]` | Не замечание — запрос разъяснения. | `[question] Why do we cache for 30 min here?` |
| `[praise]` | Явная похвала. Не игнорировать — это feedback, валидирующий подход. | `[praise] Nice extraction — makes the call site trivial.` |

### Как формулировать

- **MUST** — указать **почему**, не только «что не так». `[blocker] Mutates input` — плохо; `[blocker] Mutates input — caller in cart.ts depends on the original reference` — хорошо.
- **SHOULD** — предложить альтернативу или путь исправления, если решение неочевидно.
- **SHOULD NOT** — фокусироваться на стиле, который ловит линтер (это работа линтера, не ревьюера).
- **MUST NOT** — переходить на личности, сарказм, «это очевидно же». Feedback — про код, не про автора.

## 5. Критерии approve

Approve — сигнал «я считаю, что это готово к мёржу при условии, что остальные требования выполнены».

Ревьюер **MAY** approve, если:

- [ ] Все `[blocker]`-комментарии резолвнуты (fix или обоснованное несогласие с записью).
- [ ] Все `[issue]`-комментарии резолвнуты или переведены в follow-up issue.
- [ ] Чек-лист раздел 3 пройден в разумной степени (не формально).

Ревьюер **MUST NOT** approve, если:

- Есть нерезолвленный `[blocker]`.
- CI не зелёный.
- Описание PR не заполнено.
- Ревьюер не понял часть изменений — **MUST** оставить `[question]` вместо approve.

### «Approve с оговорками»

GitHub позволяет «Approve» с не-резолвленными `[suggestion]`/`[nit]`. Это **MAY**, если:

- Автор PR явно подтвердил, что видел комментарии.
- Ревьюер уверен, что автор не забудет их обработать.
- Нет blocker-ов.

## 6. Разногласия

Если автор PR не согласен с комментарием ревьюера:

1. **Автор** — отвечает в threads'е с обоснованием (технические аргументы, ссылка на правило, измерения).
2. **Ревьюер** — либо соглашается и resolves thread, либо эскалирует.
3. **Эскалация** — привлекается третий участник: владелец категории / tech lead / дополнительный ревьюер из той же области. Его мнение — тайбрейкер.
4. **Итог** — фиксируется в thread'е явно: «resolved: принято решение X по причине Y».

### Запрет на «продавливание»

- **MUST NOT** — автор резолвит `[blocker]` ревьюера самостоятельно, не получив согласия ревьюера или эскалации.
- **MUST NOT** — ревьюер блокирует PR бесконечно без конкретики. Если после правок ревьюер не может сформулировать конкретный `[blocker]` — **MUST** approve либо делегировать.

### Фиксация решений, меняющих правила

Если разногласие выявляет пробел в правилах — открывается ADR или issue на правку главы. Локальное решение в PR **MAY** быть принято, но **SHOULD** сопровождаться ссылкой на открытый ADR/issue.

---

## 7. Self-review

Автор PR **MUST** провести self-review перед request-review:

- Прочитать собственный diff полностью, как если бы это был чужой PR.
- Прогнать чек-лист раздел 3 — найти хотя бы один `[nit]` самому (редко не находится).
- Убедиться, что описание PR соответствует тому, что в diff'е.

Self-review занимает 5–15 минут и экономит ревьюеру 30+ минут.

---

## 8. Запрещённые паттерны

- **«LGTM» без чтения кода** — **MUST NOT**. Approve — осмысленное действие, не кнопка вежливости.
- **Ревью через 30 секунд после request** — **SHOULD NOT** без явного «я уже видел этот код в draft». Быстрое ревью либо поверхностное, либо не происходило.
- **Игнорирование `[blocker]` автором («closes as not planned»)** без ответа на суть — **MUST NOT**.
- **Эскалация в обход ревьюера** (мёрж через admin-права) — **MUST NOT** без letter-recorded причины.
- **Review без чек-листа раздел 3** — **SHOULD NOT** для нетривиальных PR. Чек-лист — не формальность, а защита от забытых проверок.
- **Feedback, не следующий формату префиксов раздел 4** — **SHOULD NOT**. Без префикса автор угадывает силу замечания.

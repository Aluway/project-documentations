# 01 — Git: ветки, коммиты, история

> Оглавление: [`README.md`](README.md).

Правила работы с git: ветвление, коммит-конвенция, атомарность изменений, обращение с публичной историей, lockfile'ы и `.gitignore`. Применяются ко всем репозиториям-потомкам этого шаблона.

---

## 1. Ветвление

### Основная ветка

- `main` — единственная защищённая ветка. Всегда в релизо-готовом состоянии.
- Push напрямую в `main` **MUST NOT**. Все изменения — через PR.
- Force-push в `main` **MUST NOT** без письменного согласия владельца репозитория и уведомления всех контрибьюторов.

### Именование feature/fix-веток

Формат: `<type>/<short-kebab-desc>`.

| Префикс | Когда |
|---|---|
| `feature/` | Новая функциональность (`feature/add-user-filter`, `feature/dark-mode-toggle`). |
| `fix/` | Исправление бага (`fix/login-redirect-loop`, `fix/date-formatting`). |
| `docs/` | Правки в `docs/**`, комментариях, README (`docs/update-architecture-v2.1`). |
| `chore/` | Инфраструктура, зависимости, конфиг (`chore/bump-typescript-5.5`). |
| `refactor/` | Рефакторинг без изменения поведения (`refactor/extract-user-store`). |
| `test/` | Только тесты (`test/add-checkout-e2e`). |
| `hotfix/` | Срочное исправление в релизе (`hotfix/v2.3.1-payment-crash`). |

- Имя после префикса — **kebab-case**, 2–5 слов, описательное.
- **MUST NOT** — `feature/ART-123` или другие JIRA-подобные коды без человеческого описания. Номер задачи **MAY** идти после описания: `feature/user-filter-ART-123`.
- **MUST NOT** — ветки без префикса (`my-branch`, `new-stuff`).

### Жизненный цикл

- Feature-ветка **SHOULD** жить ≤ 5 рабочих дней. Долгоживущие ветки — сигнал, что задача не декомпозирована.
- После мёржа — ветка **MUST** удаляться (локально и на remote).
- Stale-ветки (> 30 дней без активности) **SHOULD** закрываться владельцем репозитория.

## 2. Конвенция коммитов

Формат — вариант Conventional Commits 1.0.0, адаптированный под репо.

### Структура

```
<type>(<scope>): <subject>

<body (optional)>

<footer (optional)>
```

### Типы

| Тип | Когда |
|---|---|
| `feat` | Новая функциональность для пользователя / API. |
| `fix` | Исправление бага в существующей функциональности. |
| `docs` | Только документация: `docs/**`, JSDoc, README, комментарии. |
| `style` | Форматирование, whitespace, точки с запятой — **не меняет поведение**. |
| `refactor` | Изменение кода, не меняющее поведение и не исправляющее баг. |
| `test` | Добавление или правка тестов. |
| `chore` | Инфраструктура: зависимости, конфиг тулинга, CI. |
| `perf` | Улучшение производительности, измеряемое. |

### Правила subject

- **MUST** — длина ≤ 50 символов (включая `<type>(<scope>): `).
- **MUST** — повелительное наклонение на английском: `add`, `fix`, `remove` — не `added`, `fixes`, `removing`.
- **MUST** — нижний регистр первой буквы subject: `feat(auth): add oauth login`, не `Add OAuth login`.
- **MUST NOT** — точка в конце subject.
- **SHOULD** — конкретика: `fix(cart): handle empty basket on checkout` — не `fix: cart bug`.

### Scope

- **SHOULD** — указывать scope в скобках. Scope обычно соответствует FSD-слайсу или домену: `auth`, `cart`, `user`, `api`, `ci`, `deps`.
- **MAY** — опускать scope для коммитов, затрагивающих несколько областей (`chore: bump node to 22`).

### Body

- **MAY** — если subject недостаточен для понимания «что / почему».
- **MUST** — ширина строк ≤ 72 символа (читаемость в `git log`).
- Объясняет **что** и **почему**, не **как** (код покажет «как»).
- Отделяется от subject пустой строкой.

### Footer

- Ссылки на задачи: `Refs: #123`, `Closes #456`, `See-also: #789`.
- Breaking change: `BREAKING CHANGE: <описание>` — **MUST** для любого `feat`/`fix`, ломающего внешний API.
- Co-authors: `Co-Authored-By: <Name> <email>`.

### Примеры

✓ Корректно:
```
feat(auth): add oauth login via google

Implements OAuth 2.0 code flow. Reuses existing session
middleware; adds /api/auth/google callback and a button
to the sign-in page.

Closes #142
```

```
fix(cart): handle empty basket on checkout
```

```
refactor(user): extract avatar logic into shared/ui
```

✗ Некорректно:
```
Added new feature.                           # ✗ нет type, wrong tense, точка
fix: stuff                                   # ✗ не конкретно
feat(auth): Implements OAuth 2.0 code flow…  # ✗ Upper-case, длина > 50
```

## 3. Атомарность коммитов

- **MUST** — один коммит = одно логическое изменение. Тесты, которые верифицируют фичу, **MAY** быть в том же коммите, что и сама фича, — это одно изменение.
- **MUST NOT** — смешивать рефакторинг и фичу в одном коммите. Сначала коммит `refactor: …`, потом `feat: …` поверх.
- **MUST NOT** — смешивать форматирование и правки логики. `style:` отдельно от `fix:`/`feat:`.
- **SHOULD** — коммит проходит тесты и линтер в изоляции. Не «зависит от следующего коммита в той же ветке».

### Приёмы

- Use `git add -p` для частичного стейджинга файла — разложить смешанные правки по разным коммитам.
- Use `git rebase -i` **до открытия PR** — разбить один большой коммит на несколько атомарных, или слить серию wip-коммитов.

## 4. Публичная vs локальная история

- **Локальная история** (ветка не запушена или запушена в свою feature-ветку без открытого PR) — **MAY** переписываться: rebase, squash, amend. Это черновик.
- **Публичная история** (ветка с открытым PR, просмотренным другим; или `main`) — **MUST NOT** переписываться. Другие разработчики уже видели эти коммиты; переписывание ломает их локальные клоны и review-состояние.

### Rebase vs merge feature → main

- **SHOULD** — `rebase` feature-ветки поверх свежего `main` **перед** открытием PR: это даёт линейную историю без merge-commits.
- После открытия PR — **MUST NOT** `rebase` публично, если код уже ревьюили. Обновление — через `merge main` в feature-ветку или через перезапрос ревью (только с согласия ревьюера).
- Merge в `main` — либо **squash-merge** (для шумных веток — **SHOULD**), либо **rebase-merge** (для атомарно оформленных серий коммитов — **MAY**). Merge-commit (`--no-ff`) — **SHOULD NOT** без причины.

### Запрещённые операции

- `git push --force` в `main`, релизную ветку или чью-то чужую ветку — **MUST NOT**.
- `git push --force-with-lease` в свою feature-ветку с открытым PR — **MAY**, но **SHOULD** уведомить ревьюера перед этим.
- `git reset --hard` на публичной ветке — **MUST NOT**. Откат — через `git revert` (создаёт новый коммит, сохраняет историю).

## 5. Lockfile'ы

Зависимостный lockfile фиксирует воспроизводимость сборки. Репо **MUST** содержать ровно один lockfile, соответствующий выбранному пакетному менеджеру:

| Менеджер | Файл |
|---|---|
| npm | `package-lock.json` |
| pnpm | `pnpm-lock.yaml` |
| yarn (classic) | `yarn.lock` |
| yarn (berry) | `yarn.lock` + `.yarn/` |

- Lockfile **MUST** коммититься в git.
- **MUST NOT** иметь в репо два lockfile'а разных менеджеров одновременно.
- Правки lockfile'а — в том же коммите, что и правки `package.json`, вызвавшие обновление. **MUST NOT** — отдельный коммит «update lockfile» без сопутствующего `package.json`-изменения.

## 6. `.gitignore` и что не коммитить

Эта глава регулирует git-сторону: что не попадает в историю и через `.gitignore`. Полная политика обращения с секретами — [`../security/01-secrets.md`](../security/01-secrets.md).

- **MUST NOT** коммитить: `.env`, `.env.local`, `.env.*.local`, credential-файлы, ключи SSH/GPG, токены OAuth, сертификаты.
- **MUST NOT** коммитить: build-output (`dist/`, `build/`, `.next/`, `coverage/`), логи (`*.log`), IDE-кэш (`.vscode/settings.json` — пограничный; команда **MAY** явно коммитить shared-settings).
- **MUST** — в репо лежит `.env.example` / `.env.sample` с плейсхолдерами реальных переменных и комментариями, откуда их брать (детали контракта — [`../security/01-secrets.md`](../security/01-secrets.md) раздел 3).
- Если секрет всё-таки попал в историю — **MUST** выполнять процедуру из [`../security/01-secrets.md`](../security/01-secrets.md) раздел 5 (ротация → чистка истории → уведомление → post-mortem). Простое удаление файла в следующем коммите **MUST NOT** считаться исправлением — секрет остаётся в истории и виден через `git log -p`.

## 7. Запрещённые паттерны

- **Коммит `wip`, `fix`, `ok`, `asdf` и т.п. без type/scope/subject** — **MUST NOT**. Даже локально — используйте `git commit --fixup` или ветку-черновик, но в PR такого быть не должно.
- **Коммит, в котором тесты красные** — **MUST NOT**. Коммит **MUST** проходить тесты и линтер в изоляции.
- **Коммит, сгенерированный автоматически без авторства** — **MUST NOT**. Автор коммита — человек или явно обозначенный агент, отвечающий за правильность.
- **Смешанные коммиты с сотнями файлов** (например, «big refactor») — **SHOULD NOT**. Разбивайте на серию.
- **Merge-commits в feature-ветку без явной необходимости** — **SHOULD NOT**. Синхронизация с `main` — через rebase.
- **Переписывание публичной истории** — **MUST NOT**. См. раздел 4.
- **Игнорирование hooks (`--no-verify`, `--no-gpg-sign`) без явного разрешения owner'а** — **MUST NOT**. Провал hook — сигнал о проблеме; обходить её шорткатом **MUST NOT**.

# 02 — Pull Requests

> Оглавление: [`README.md`](README.md).

Правила открытия, оформления и ведения PR: размер, заголовок, шаблон описания, draft-режим, связь с issues и ADR.

---

## 1. Размер PR

- **SHOULD** — diff ≤ 500 строк добавлено+удалено. Это ориентир, подтверждённый исследованиями эффективности ревью.
- **MUST** — если diff > 1000 строк и PR не помечен как «pure generated» (миграция, автоформат, codemod), — PR **MUST** быть разбит на серию либо в описании **MUST** быть обоснование, почему разбить невозможно.
- Forced-длинные PR (лёгитимные исключения): генерированные файлы (snapshots, types-from-openapi), массовые переименования, первичная миграция.

### Как разбивать

- По слоям FSD: сначала PR с `shared/` / `entities/`, затем поверх — `features/` / `pages/`.
- По этапам рефакторинга: сначала «подготовительный» PR (extract helper, add tests), затем «основной» (собственно рефакторинг).
- Один PR — одна тема. Если в PR есть «попутно я ещё пофиксил X» — X выносится в отдельный PR.

## 2. Заголовок PR

Формат — тот же, что и commit subject:

```
<type>(<scope>): <subject>
```

- Правила — те же: ≤ 50 символов, повелительное наклонение, lower-case, без точки в конце. См. [`01-git.md`](01-git.md) раздел 2.
- Если PR содержит один коммит — заголовок PR **SHOULD** совпадать с subject этого коммита.
- Если PR — серия коммитов — заголовок описывает результирующее изменение на более высоком уровне.

✓ `feat(cart): add promo code redemption`
✓ `refactor(user): migrate avatar to shared/ui`
✗ `Various improvements` (нет type, нет конкретики)
✗ `[ART-123] Update stuff` (нет type, не описательно)

---

## 3. Шаблон описания PR

**MUST** заполнять каждую секцию; пустые — **MUST NOT** оставлять (кроме Screenshots, если PR не UI).

```markdown
## Summary

<1–3 буллета: что меняется на уровне продукта / API. Без деталей реализации.>

## Why

<Мотивация: проблема, ссылка на issue / ADR / request. 1–3 предложения.>

## Changes

<Детали реализации, заслуживающие внимания ревьюера. Технические решения, trade-off'ы, альтернативы.>

## Test plan

- [ ] <Сценарий, который был проверен>
- [ ] <Другой сценарий>
- [ ] Unit/integration tests added / updated
- [ ] Manual smoke test passed

## Screenshots / Recordings

<Для UI-изменений — скриншоты «до / после» или запись. Для backend — можно опустить.>

## Related

- Closes #<issue>
- Refs #<issue>
- Implements [ADR NNNN](../docs/decisions/NNNN-...)
```

### Правила для секций

- **Summary** — 1–3 буллета. Ровно то, что войдёт в changelog / release notes.
- **Why** — обязательно; PR без мотивации **MUST NOT** приниматься.
- **Changes** — техническая часть. Не пересказ diff'а: ревьюер сам прочитает код. Нужны *решения*, *альтернативы*, *не-очевидности*.
- **Test plan** — чек-лист того, что было проверено. **MUST** содержать хотя бы один пункт. Пустой test plan — **MUST NOT**.
- **Related** — ссылки на issue и ADR. Если PR реализует ADR с `status: accepted` — **MUST** быть ссылка.

---

## 4. Draft PRs

- **MAY** — для WIP-работы, чтобы получить раннюю обратную связь по подходу, не по коду.
- **MAY** — для демонстрации approach'а перед полной реализацией.
- **SHOULD** — в описании draft-PR в начале указать, что именно нужно от ревьюеров: «Approach review only, implementation details WIP» или «Review only X, остальное — stub».
- Draft-PR **MUST NOT** мёржиться. Перед мёржем — перевод в «Ready for review» и получение требуемых approve.

---

## 5. Связь с issues и ADR

### Issues

- **SHOULD** — каждый нетривиальный PR связан с issue через `Closes #N` / `Refs #N`. Тривиальные (typo в doc) — **MAY** без issue.
- `Closes #N` в описании или коммите закрывает issue автоматически при мёрже в `main`.

### ADR

- Если PR реализует ADR — **MUST** быть ссылка в секции Related.
- Если PR **сам** принимает ADR (добавляет файл в `docs/decisions/` со `status: accepted`) — заголовок PR **SHOULD** иметь scope `adr`: `docs(adr): accept ADR 0008 — ...`.
- Если PR вводит новое структурное решение (новая категория, смена frontmatter-схемы) — ADR **MUST** быть в том же PR. Правило из [`../_meta/governance.md`](../_meta/governance.md) раздел 4.

---

## 6. Обновление PR после ревью

- Правки после ревью — новыми коммитами поверх, **MUST NOT** squash/amend публичной истории до мёржа. Ревьюер должен видеть, что именно изменилось в ответ на его комментарии.
- При мёрже (squash-merge) финальный коммит собирается автоматически — промежуточные коммиты не остаются в `main`.
- **MUST NOT** резолвить conversation ревьюера «сам за себя» без его подтверждения. Resolve — прерогатива автора комментария.
- **MUST** — отвечать на каждый **blocker**-комментарий: либо правкой, либо явным «не согласен, потому что X, закрываю как не принятое» с обоснованием.

## 7. Мёрж

### Условия мёржа

PR **MUST** иметь:

- [ ] Зелёный CI (все обязательные проверки прошли).
- [ ] Нужное число approve (см. [`03-code-review.md`](03-code-review.md) раздел 1).
- [ ] Все blocker-комментарии резолвнуты автором комментария или заменены более свежим ревью.
- [ ] Описание PR заполнено по шаблону раздел 3.
- [ ] Test plan выполнен (чек-бокс отмечен или есть объяснение).
- [ ] Конфликтов с `main` нет.

### Стратегия мёржа

- **Squash merge** — дефолт для большинства PR. Серия wip-коммитов превращается в один чистый коммит в `main`. Subject squash-коммита = заголовок PR.
- **Rebase merge** — для PR с атомарно оформленной серией коммитов, где каждый коммит осмыслен и проходит тесты. Применяется, когда автор явно структурировал историю.
- **Merge commit** (`--no-ff`) — **SHOULD NOT** без причины.

### Кто мёржит

- **SHOULD** — мёржит автор PR после получения approve и зелёного CI. Это закрывает feedback loop: «я открыл, я довёл, я мёрджу».
- **MAY** — мёржит ревьюер, если явно договорились («approve + merge once CI green»).
- **MUST NOT** — мёржить чужой PR без согласия автора, кроме случаев автоматической очистки (автор недоступен > 7 дней, PR блокирует другие).

---

## 8. После мёржа

- Ветка **MUST** быть удалена (локально и на remote). Большинство git-хостингов делают это автоматически — опция **SHOULD** быть включена для репо.
- Связанные issue **MUST** быть проверены на фактическое закрытие (если `Closes #N` сработал — ок; иначе — закрыть вручную).
- Если PR реализовал ADR — `status` ADR **MUST** перейти из `proposed` в `accepted`, если ещё не был. Финальный accept **MAY** происходить отдельным PR или в самом merge-коммите.

---

## 9. Запрещённые паттерны

- **PR без описания или с описанием «see commits»** — **MUST NOT** приниматься.
- **PR, в котором несколько не-связанных тем** («fix A, feat B, refactor C in one PR») — **MUST NOT**. Разделять.
- **Мёрж без approve** (self-approve, кроме оговорённых случаев repo без reviewers) — **MUST NOT**.
- **Мёрж с красным CI**, даже «мы знаем, это не связано с PR» — **MUST NOT**. Сначала чинится CI, потом мёрж.
- **Skipping required checks через admin-merge** — **MUST NOT** без письменного разрешения owner'а в описании PR.
- **PR от бота без human review** — **MUST NOT** для изменений логики. Чисто-авто-PR (dependency bumps) — **MAY** с авто-approve, если настроены гейты безопасности.

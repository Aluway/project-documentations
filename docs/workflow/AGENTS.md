# AGENTS.md — Workflow (git, PR, review, releases)

> Категорийная точка входа для ИИ-агентов.
> Главы написаны на русском. Покрывают процессную часть разработки: как ветвиться, как коммитить, как открывать и ревьюить PR, как релизиться.

---

## Scope

Эта категория покрывает **стек-агностичный процесс разработки**:

- Правила ветвления и работы с git-историей.
- Конвенция коммитов и атомарность изменений.
- Размер PR, заголовок, шаблон описания, связка с issues.
- Протокол code-review: кто ревьюит, SLA, чек-лист, формы feedback.
- Релизный цикл: semver, changelog, tagging, hotfix.

**Не покрывает**:

- Архитектурные правила размещения кода — [`../architecture/AGENTS.md`](../architecture/AGENTS.md).
- Стиль кода (TS, React, формы, стили, тесты) — [`../code-style/AGENTS.md`](../code-style/AGENTS.md).
- Язык коммит-сообщений (регулируется глобально) — [ADR 0006](../decisions/0006-language-policy.md).

## Working Protocol

Для задачи, затрагивающей git-процесс или релизный цикл:

1. **Ветвление** — перед началом работы убедитесь, что ветка именована по [`01-git.md`](01-git.md) раздел 1.
2. **Коммиты** — по ходу работы коммитьте атомарно; формат subject — [`01-git.md`](01-git.md) раздел 2.
3. **PR** — перед открытием прогоните [`02-pull-requests.md`](02-pull-requests.md): размер, заголовок, шаблон.
4. **Review** — если вы ревьюер, работайте по [`03-code-review.md`](03-code-review.md); если автор — учитывайте чек-лист оттуда при правке.
5. **Релиз** — на этапе релиза — [`04-releases.md`](04-releases.md): semver-решение, changelog, tag.
6. **Верификация перед финишем** — Pre-Flight Checklist ниже плюс [Global Pre-Flight](../../AGENTS.md#global-pre-flight).

## Hard Invariants

- Имя ветки — kebab-case с префиксом типа (`feature/`, `fix/`, `docs/`, `chore/`, `hotfix/`) — **MUST**. См. [`01-git.md`](01-git.md) раздел 1.
- Commit subject следует формату `<type>(<scope>): <subject>` — **MUST**. Типы из [`01-git.md`](01-git.md) раздел 2. Длина subject — ≤ 50 символов, в повелительном наклонении, без точки в конце.
- Один коммит — одно логическое изменение — **MUST**. Смешанные коммиты (рефакторинг + фича + форматирование) **MUST NOT** приниматься.
- `main` — защищённая ветка; push напрямую **MUST NOT**. Все изменения — через PR.
- Публичная история (коммиты, попавшие в `main` или в PR, открытый другим) **MUST NOT** переписываться (`--force`, `rebase -i`). Локальная история до открытия PR — **MAY** переписываться.
- Секреты, `.env`, credentials **MUST NOT** попадать в git-историю (см. [`01-git.md`](01-git.md) раздел 6).
- PR-заголовок следует тому же формату, что и commit subject — **MUST**. См. [`02-pull-requests.md`](02-pull-requests.md) раздел 2.
- Связанный ADR **MUST** быть упомянут в описании PR, если PR реализует решение ADR.
- Semver-тег на релизной точке имеет префикс `v` (`v1.2.3`) — **MUST**. См. [`04-releases.md`](04-releases.md) раздел 3.

## Pre-Flight Checklist

- [ ] Имя ветки соответствует [`01-git.md`](01-git.md) раздел 1.
- [ ] Все коммиты следуют формату `<type>(<scope>): <subject>` — длина, наклонение, отсутствие точки в конце.
- [ ] Коммиты атомарны; нет смешанных (рефакторинг + фича, или изменение + форматирование).
- [ ] В коммитах нет секретов, `.env`, credentials.
- [ ] PR-заголовок соответствует формату; описание заполнено по шаблону из [`02-pull-requests.md`](02-pull-requests.md) раздел 3.
- [ ] Если PR реализует ADR — ADR упомянут в описании и имеет `status: accepted`.
- [ ] Если PR больше 500 строк diff — есть обоснование в описании или PR разделён.
- [ ] Для релизного PR — обновлены версия, changelog, tag (см. [`04-releases.md`](04-releases.md)).

# Style Guide — как писать главы документации

> Оглавление: [`AGENTS.md`](AGENTS.md).

Правила для авторов и ИИ-агентов, правящих документацию. Применяются ко всем файлам в `docs/**/*.md`, кроме `README.md` категорий (там допустим более свободный описательный стиль).

---

## 1. Язык

- **Русский** — содержательные главы внутри категорий (`docs/<category>/**/*.md`), включая `AGENTS.md` категорий.
- **Английский** — корневой [`../../AGENTS.md`](../../AGENTS.md), имена файлов, код-примеры, идентификаторы, значения frontmatter.
- **MUST NOT** смешивать языки в пределах одного абзаца. Технические термины в русском тексте — латиницей без перевода (props, Public API, tree-shaking).

---

## 2. RFC 2119 keywords

Правила формулируются через заглавные keywords — это единственный источник «силы» требования.

| Keyword | Сила | Пример |
|---|---|---|
| `MUST` / `REQUIRED` | Обязательно. Нарушение — блокер ревью. | «`index.ts` **MUST** содержать явные именованные экспорты.» |
| `MUST NOT` | Запрет. | «`any` **MUST NOT** использоваться вне границ с нетипизированными библиотеками.» |
| `SHOULD` / `RECOMMENDED` | Настоятельно рекомендуется. Отклонение возможно с обоснованием в PR. | «Server state **SHOULD** жить в TanStack Query.» |
| `SHOULD NOT` | Настоятельно не рекомендуется. | «Non-null assertion (`!`) **SHOULD NOT** использоваться без обоснования.» |
| `MAY` / `OPTIONAL` | Допустимо. | «Для мутаций с задержкой > 100 мс **MAY** использоваться optimistic updates.» |

### Запрещённые формулировки

Слова-паразиты, снимающие проверяемость правила, **MUST NOT** появляться в месте, где уместен RFC 2119 keyword:

- «желательно», «по возможности», «стоит», «лучше», «предпочтительнее» — замените на `SHOULD` / `SHOULD NOT`.
- «нельзя» — замените на `MUST NOT`.
- «нужно», «надо», «требуется» — замените на `MUST` / `REQUIRED`.
- «можно» — замените на `MAY` (если действительно опционально) или `SHOULD` (если рекомендуется).

Keywords пишутся **заглавными** и **жирным** (`**MUST**`). В тексте — `**MUST**`, не _MUST_ и не `MUST` без выделения.

---

## 3. Структура главы

Главы в `universal/` и `variants/` следуют шаблону (см. [`templates/chapter-universal.md`](templates/chapter-universal.md) / [`templates/chapter-variant.md`](templates/chapter-variant.md)):

1. **Frontmatter** — метаданные (см. [`frontmatter.md`](frontmatter.md)).
2. **Заголовок** — `# NN — Название (universal|variant)`, где `NN` — двузначный номер.
3. **Блок-цитата с навигацией** — ссылка на оглавление категории и, при необходимости, на `PROFILE.md`.
4. **Для variant** — явное условие активации: «**Активен, если в `PROFILE.md` <поле> = <значение>.**»
5. **Короткий intro-абзац** — что эта глава регулирует, за 1–3 предложения.
6. **Нумерованные разделы** — `## 1. Название`, `## 2. ...`.
7. **Антипаттерны** — отдельный раздел в конце главы с `MUST NOT` / `SHOULD NOT`.

ADR — собственный шаблон (см. [`templates/ADR.md`](templates/ADR.md)).

`AGENTS.md` категории — фиксированные секции: Scope / When to Read / Working Protocol / Hard Invariants / Pre-Flight Checklist / «If Chapters Disagree».

---

## 4. Длина и детализация

- Глава **SHOULD** укладываться в 300 строк. Больше — сигнал, что тема дробится на две.
- Один раздел главы **SHOULD** быть не длиннее 80 строк.
- Каждое правило **MUST** формулироваться за 1–2 предложения. Объяснение «почему» (rationale) — сноской или отдельной цитатой; длинное обоснование выносится в ADR.

---

## 5. Ссылки

### Markdown-ссылки, не ссылки текстом

- ✓ `см. [`../code-style/universal/01-typescript.md`](../code-style/universal/01-typescript.md)`
- ✗ «см. файл `code-style/universal/01-typescript.md`»

### Относительные пути

- Ссылки между главами одной категории — относительные (`[./02-slices.md]`).
- Ссылки между категориями — относительные через `../` (`[../code-style/AGENTS.md]`).
- Ссылки на код репозитория (если бы появился) — через корневой алиас (`[src/shared/ui]`) в соответствии с правилами IDE-интеграции.

### Якоря

- На конкретный раздел — `[текст](path.md#заголовок-в-kebab-case)`. Заголовки русского текста якорятся как есть (транслитерация автоматически в большинстве рендереров; если рендерер упирается — сверяйтесь с фактическими якорями).

### Битые ссылки

**MUST NOT** коммитить главу с битой внутренней ссылкой. При переименовании главы — обновите все упоминания в том же коммите.

## 6. Code-примеры

- **MUST** иметь указание языка в fence для блоков с кодом реального языка: ` ```ts`, ` ```tsx`, ` ```json`, ` ```bash`, ` ```css`, ` ```yaml`, ` ```dotenv`.
- **MAY** использовать `text` или **bare** ` ``` ` fences для блоков, не являющихся кодом: директорные деревья, ASCII-диаграммы, вывод консоли, plain-text шаблоны (commit subject примеры).
- **MUST** быть минимально компилируемым фрагментом (для кода). Псевдокод — только если явно подписан как псевдокод.
- Комментарии внутри примеров — на **английском** (согласно языковой политике кода).
- Маркеры корректности — `✓ Корректно:` перед блоком кода, `✗ Некорректно:` или `✗ Анти-паттерн:` перед блоком нарушения.
- Отступы: 2 пробела в TS/TSX/JSON. Без табов.

---

## 7. Таблицы

Используются для:
- When to Read — «файл / когда открывать».
- Compatibility Matrix — «значение A / значение B / требование / конфликт».
- Сравнение вариантов.

**SHOULD NOT** использоваться для больших списков правил — правила лучше читаются как нумерованные разделы с bullet-списками.

---

## 8. Чек-листы

- Pre-Flight Checklist в `AGENTS.md` категории **MUST** быть коротким (7–15 пунктов) и содержать только проверяемые утверждения.
- Длинные чек-листы по типам задач — в отдельной главе (например, [`../architecture/07-checklists.md`](../architecture/07-checklists.md)).
- Каждый пункт чеклиста **MUST** начинаться с `- [ ]` и формулироваться в **результативной форме** («Код размещён по pages-first flow»), не в процессной («Положить код по pages-first flow»).

---

## 9. Анти-паттерны (в документации)

- **MUST NOT** дублировать правило между главами. Дубль — ссылка.
- **MUST NOT** писать правило в `AGENTS.md` без его подробной формулировки в главе. `AGENTS.md` — выжимка.
- **MUST NOT** использовать «мы» / «наш проект» / «наша команда» — универсальная документация пишется в безличной форме или во 2-м лице («Применяйте…», «Ставьте…»).
- **MUST NOT** оставлять TODO в теле главы без пометки в `status: draft` во frontmatter.
- **SHOULD NOT** вставлять скриншоты и ASCII-диаграммы, если ту же информацию можно передать таблицей или списком — агент парсит текст, не картинки.
- **SHOULD NOT** цитировать внешние источники (Wikipedia, блог-посты) как авторитет. Ссылка на первоисточник — `README.md` категории в секции «Официальные источники».

---

## 10. Типографика

- Длинное тире — `—` (не `--`, не `-`). В Windows-окружении вставляется через clipboard; в агентских генерациях — прямо символ.
- Кавычки — «» в русском тексте, `"` в коде и frontmatter.
- Списки с bullet — `-`, не `*`.
- Нумерованные списки — `1.`, `2.` (markdown сам нумерует, но для читаемости raw-текста соблюдайте последовательность).

### Ссылки на разделы

- При ссылке на раздел главы **MUST** использовать русское слово «раздел»: `[`foo.md`](foo.md), раздел 2`, `[`foo.md`](foo.md) — раздел 2`, `(см. [`foo.md`](foo.md), раздел 2)`.
- Знак параграфа `§` **MUST NOT** использоваться — он формально-юридический, диссонирует с русским техническим текстом и создаёт визуальный шум в markdown-рендере. Это распространяется на все главы, ADR, `AGENTS.md` и шаблоны.
- Если раздел имеет уникально-читаемый заголовок (`Fallback-стратегия`, `Compatibility Matrix`) — **MAY** ссылаться по имени: `[Fallback-стратегия](...)` вместо `раздел 5`.

# Governance — владение и жизненный цикл документации

> Оглавление: [`AGENTS.md`](AGENTS.md).

Описывает, кто отвечает за каждую категорию, когда главы ревьюятся, как депрецируются и как меняется структура. Без этих правил универсальная документация со временем расходится между репозиториями и теряет авторитет.

---

## 1. Владение (Ownership)

### Категории

Каждая категория **MUST** иметь ответственного владельца. Владелец — команда или роль (не человек, чтобы не ломалось при кадровых изменениях).

| Категория | Рекомендуемый владелец |
|---|---|
| `_meta` | Команда, ведущая стандарты документации (например, platform / dx). |
| `architecture` | Команда, владеющая архитектурой фронтенда. |
| `code-style` | Та же команда или frontend-guild. |
| `decisions` | Коллективно — каждая ADR имеет автора; категория в целом без единого владельца. |

Фактическое имя владельца фиксируется одним из способов, в порядке приоритета:

1. `CODEOWNERS` репозитория — **SHOULD** использоваться как основной источник.
2. Поле `owner:` во frontmatter'е категорийного `AGENTS.md` — **MAY** использоваться, если `CODEOWNERS` отсутствует.
3. Явная таблица в этом файле — **MAY** использоваться как fallback.

### Полномочия владельца

Владелец категории:

- **MUST** утверждать изменения в `universal/*` своей категории.
- **MUST** проводить плановое ревью (см. раздел 2) и обновлять `last-reviewed`.
- **SHOULD** утверждать новые главы в `variants/*`.
- **MAY** делегировать правки в примерах и формулировках без формального ревью.

Структурные решения (новая категория, переименование слоя/сегмента, смена шаблона, смена frontmatter-схемы) **MUST** оформляться ADR в [`../decisions/`](../decisions/AGENTS.md) и проходить через согласование владельцев всех затронутых категорий.

## 2. Каденс ревью

| Артефакт | Частота планового ревью | Триггеры внепланового ревью |
|---|---|---|
| `universal/*.md` | Раз в 6 месяцев | Breaking-изменение в стеке; ADR, затрагивающий правило. |
| `variants/*.md` | При выходе мажорной версии соответствующей библиотеки | Обновление минимальной версии в `requires.min`; смена API библиотеки. |
| `PROFILE.md` | При каждом PR, меняющем стек | Обновление версии любой зависимости из `Core` / `Compiler` / `State` / `Forms` / `Styling` / `Testing` / `Tooling`. |
| `AGENTS.md` категории | При добавлении/удалении главы в её When to Read | Изменение Pre-Flight, Hard Invariants. |
| `_meta/*.md` | Раз в 12 месяцев | Смена шаблона или frontmatter-схемы — через ADR. |
| `decisions/*.md` | Не ревьюятся (append-only) | Решение заменяется новым ADR через `supersedes`. |

Плановое ревью — даже без правки содержимого — **MUST** сопровождаться bump'ом `last-reviewed` во frontmatter (формат `YYYY-MM-DD`, см. [`frontmatter.md`](frontmatter.md)).

## 3. Жизненный цикл главы

```
draft  →  active  →  deprecated  →  удалена
             ↓             ↑
         (ревью,      (новый ADR
          без смены    заменяет
          статуса)     главу)
```

### `draft`

- Новая глава, правила ещё не окончательны.
- Агент **MAY** применять правила, но **MUST** пометить вывод: «применено draft-правило из <путь>, требуется ревью».
- Срок в `draft` **SHOULD NOT** превышать 3 месяца. По истечении — либо `active`, либо удаление.

### `active`

- Правила применяются без оговорок.
- Bump `version` по semver при каждом изменении правил (см. [`frontmatter.md`](frontmatter.md) раздел 2).

### `deprecated`

- Глава оставлена для справки (ссылки из других глав могут быть), но её правила **MUST NOT** применяться к новому коду.
- В теле главы **MUST** быть секция `## Superseded by` со ссылкой на замену.
- Во frontmatter **MUST** быть поле `superseded-by` с ID ADR (или путём к заменяющей главе).
- Срок в `deprecated` **SHOULD** — один релизный цикл (квартал), далее — удаление.

### Удаление

- Удаление главы **MUST** сопровождаться ADR («Remove chapter X»), фиксирующим причину и дату.
- Все ссылки на удалённую главу **MUST** быть перенаправлены на замену или удалены в том же PR.

## 4. Процесс изменений

### Изменение в `universal/*`

1. Автор открывает PR с описанием: что меняется, почему, какие `variants/*` затронуты.
2. Если правило breaking (меняет `MUST` ↔ `SHOULD`, удаляет правило) — **MUST** быть ADR.
3. Согласование владельца категории.
4. Bump `version` по semver, обновление `last-reviewed`.
5. Обновление связанных `variants/*` и `AGENTS.md` — **в том же PR**.

### Добавление нового `variants/*`

1. Автор создаёт файл по шаблону [`templates/chapter-variant.md`](templates/chapter-variant.md).
2. Добавляет поле в [`../code-style/PROFILE.md`](../code-style/PROFILE.md) и чекбокс в «Active Variant Modules».
3. Обновляет [`../code-style/AGENTS.md`](../code-style/AGENTS.md) — секции When to Read — Variants и Compatibility Matrix (если введены новые конфликты).
4. Без ADR, если вариант покрывает уже обсуждённый инструмент (добавили alternative state-менеджер рядом с существующими).
5. С ADR, если вариант вводит новый домен (например, первая глава по i18n).

### Добавление новой категории `docs/<category>/`

1. ADR: «Introduce category <name>».
2. Создание `docs/<category>/AGENTS.md` по шаблону [`templates/AGENTS-category.md`](templates/AGENTS-category.md).
3. Создание `docs/<category>/README.md` — описательное оглавление.
4. Обновление корневого [`../../AGENTS.md`](../../AGENTS.md) — секция Document Map.
5. Согласование владельцев всех существующих категорий (изменение коллективной точки входа).

### Изменение frontmatter-схемы

1. ADR в [`../decisions/`](../decisions/AGENTS.md).
2. Обновление [`frontmatter.md`](frontmatter.md).
3. Миграция всех существующих файлов — **в том же PR** (или в следующем, зафиксированном в том же ADR).
4. Обновление CI-линтера (если есть).

## 5. Конфликт правил и приоритет

При рассинхроне между категориями или между версиями:

1. **Свежий ADR** `accepted` — побеждает любую главу, противоречащую ему. Глава **MUST** быть обновлена в том же PR, где принимается ADR.
2. **Категорийная глава** (`docs/<category>/<chapter>.md`) — авторитетна в пределах своего scope.
3. **Категорийный `AGENTS.md`** — уступает главе при конфликте. Это подтверждено в секции «If Chapters Disagree» каждого `AGENTS.md`.
4. **Корневой `AGENTS.md`** — авторитетен для репо-wide концернов (language policy, коммит-конвенция). По категорийным — уступает категории.
5. **`_meta/*`** — авторитетен для **формы** документов, не для их содержания.

---

## 6. Версионирование категории как целого

Semver'ится **каждая глава индивидуально**, не категория целиком. Категория не имеет версии — у неё есть состав глав, каждая со своей версией.

Если нужно зафиксировать «срез» документации на момент релиза продукта — это делается через git-tag репозитория, не через поле версии документации.

---

## 7. Переиспользование между репозиториями

Если `universal/*` и `_meta/*` выносятся в общий репозиторий (см. принцип «extraction, не generalization»):

- Общий репо — `org/docs-standards` (или аналог).
- Подключается как git submodule / subtree / npm-package в каждый проект.
- `PROFILE.md`, `variants/*` (отмеченные как активные), `decisions/*` — **MUST** оставаться в каждом проекте (они специфичны).
- Обновление `universal/*` в общем репо — по процессу раздел 4 с согласованием всех заинтересованных проектов.
- В проектном репо **MUST NOT** править подключённые общие главы. Только через PR в общий репо.

До появления второго проекта вынос преждевременен — [раздел 7 выше](#7-переиспользование-между-репозиториями) описывает целевое состояние, не текущее.

## 8. Ответственность агента

ИИ-агент, правя документацию:

- **MUST** обновить `last-reviewed` при любой правке.
- **MUST** поднять `version` по semver — не «всегда PATCH», а соответствующе изменению (см. [`frontmatter.md`](frontmatter.md) раздел 2).
- **MUST** пометить в выводе, если статус главы — `draft` или `deprecated`.
- **MUST** открыть ADR при структурных изменениях (раздел 4).
- **SHOULD** отказаться применять главу с `last-reviewed` старше 24 месяцев без явного указания разработчика.
- **MUST NOT** удалять главу в том же PR, где на неё ещё остаются ссылки из других файлов.

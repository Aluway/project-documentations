# ADR 0009: Исключение i18n из scope'а template-репо

> Оглавление: [`README.md`](README.md).

---

## Status

accepted

---

## Context

Интернационализация (i18n) — извлечение user-facing-строк, плюрализация, форматирование дат/чисел по локали, RTL — **не универсальное** требование React+TS проектов. Значительная часть продуктов — одноязычные: внутренний инструментарий, B2B SaaS в одной стране, MVP-стадии. Для них правила i18n — мёртвый груз.

При этом i18n-правила существенно библиотеко-специфичны: ICU MessageFormat в react-intl, nested keys + interpolation в i18next, Server Components-aware API в next-intl. Общих «universal»-правил, полезных в каждом стеке, очень мало (извлечение в ключи, запрет конкатенации переводов) — их масса не оправдывает отдельную главу в шаблоне.

На момент решения в репо есть несколько упоминаний i18n как **фоновой концепции**:

- [`../architecture/01-layers.md`](../architecture/01-layers.md), описание слоя `app` — i18n упомянут как пример того, что живёт в `app/` (глобальный провайдер). Архитектурный factoid, не мандат.
- [`../code-style/universal/05-forms-principles.md`](../code-style/universal/05-forms-principles.md) раздел 3 — conditional guidance: «если используете i18n — кастомные messages в схеме валидации».
- [`0006-language-policy.md`](0006-language-policy.md) раздел Decision — «User-facing strings на английском до подключения i18n-решения».
- [`../_meta/governance.md`](../_meta/governance.md) раздел 4 — i18n упомянут как **пример** нового домена, требующего ADR при добавлении.

Эти упоминания не прескриптивные и не создают i18n-контракта для форка; они лишь обозначают i18n как будущий вариант расширения.

Нужно зафиксировать: template **не содержит** i18n-глав и не планирует их. Проекты-потомки, нуждающиеся в i18n, добавляют его сами.

---

## Decision

Принято следующее:

- Шаблонный репозиторий **MUST NOT** содержать глав по i18n в [`../code-style/universal/`](../code-style/universal/) и [`../code-style/variants/`](../code-style/variants/).
- Поле `i18n-library` (или аналог) **MUST NOT** фигурировать в [`../code-style/PROFILE.md`](../code-style/PROFILE.md) template-репо.
- Существующие упоминания i18n как **архитектурных** или **conditional** hint'ов (в 01-layers, 05-forms-principles, 0006-language-policy, governance) **MAY** сохраняться — они не создают i18n-контракта, только описывают место, куда бы i18n встал, если появится.
- Проект-потомок, подключающий i18n, **MUST** оформить это следующим образом:
  1. Открыть собственный ADR «Introduce i18n via <library>» со `status: accepted`.
  2. Добавить `docs/code-style/universal/11-i18n.md` со стек-агностичными правилами (извлечение, плюрализация как обязательно рассмотренный случай, запрет конкатенации переводов, namespacing ключей).
  3. Добавить `docs/code-style/variants/i18n-<library>.md` под выбранную библиотеку (react-intl / i18next / next-intl / другое).
  4. Добавить поле `i18n-library` в `PROFILE.md` и соответствующий чекбокс в «Active Variant Modules».
  5. Обновить `docs/code-style/AGENTS.md` секцию «When to Read — Variants» и Compatibility Matrix.
- Этот ADR **MUST NOT** интерпретироваться как запрет i18n в проектах-потомках. Он только фиксирует, что template не навязывает i18n.

---

## Consequences

### Positive

- Template остаётся легче на одну категорию правил. Форки, не нуждающиеся в i18n (большинство), не читают и не игнорируют нерелевантные главы.
- Чёткий механизм добавления через стандартный ADR + variant-паттерн — никакой «неформально принятой» i18n-практики.
- Архитектурные и форм-hint'ы сохраняются — они полезны и не требуют от форка выбора библиотеки.

### Negative

- Форк с i18n обязан сам написать главу и variant. Первый форк в организации, который это делает, несёт полную стоимость; последующие — копируют.
- Если через некоторое время окажется, что 80%+ форков пишут одинаковые правила по react-intl — это сигнал к пересмотру этого ADR и включению главы в template (тогда — новый ADR, `supersedes: 0009`).

### Neutral

- Решение не влияет на текущую документацию — оно отражает статус-кво и формализует его.

---

## Alternatives considered

- **Включить i18n как опциональный variant с `requires.profile.i18n-library`.** Отклонено: даже отключённый variant требует mental overhead при чтении `PROFILE.md` и `AGENTS.md` — выбор «какая библиотека» становится обязательным решением при форке, даже если ответ «нет, не используем».
- **Написать только `universal/11-i18n.md` без variants.** Отклонено: универсальные правила (извлечение в ключи, плюрализация) — почти всё, что можно сказать без специфики библиотеки; две страницы правил, которые не закрывают реальные вопросы пользователей. Variants необходимы, а значит и universal без variants бессмыслен.
- **Включить один «base» variant — react-intl как индустриальный стандарт.** Отклонено: навязывает выбор, который в Next.js-проектах неправилен (там next-intl де-факто), а в не-React-проектах инерционно; формирует bias в пользу одной библиотеки.
- **Написать заглушку «i18n category in template: TODO, see forks»** в AGENTS.md. Отклонено: TODO в template — анти-паттерн; статус категории должен быть определённым (`active` / `deprecated` / отсутствует).
- **Удалить все упоминания i18n** (в 01-layers, 05-forms, ADR 0006, governance). Отклонено: архитектурные и conditional-hint'ы не создают i18n-контракта, они описывают, **куда** i18n встанет, когда появится. Удаление лишает форки полезной ориентации.

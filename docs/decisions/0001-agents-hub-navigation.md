# ADR 0001: Двухуровневая навигация `AGENTS.md` как точка входа для агентов

> Оглавление: [`README.md`](README.md).

---

## Status

accepted

---

## Context

Документация проекта создаётся в первую очередь для ИИ-агентов, выполняющих задачи кодирования. Типичный агент начинает работу без знания структуры репо, имеет ограниченный контекст и не может эффективно «прочитать всё». Нужна навигация, которая:

1. Даёт агенту одну гарантированную точку входа.
2. Позволяет маршрутизировать задачу к нужной главе без чтения всей документации.
3. Масштабируется на произвольное число категорий.
4. Отделяет навигацию от содержательных правил — чтобы правки в правилах не ломали навигацию и наоборот.

Рассматривались альтернативы (см. ниже). Выбрана двухуровневая схема `AGENTS.md`: корневой + категорийный.

---

## Decision

Принято следующее:

- В корне репозитория **MUST** лежать `AGENTS.md` — точка входа для ИИ-агентов. Он содержит Document Map (таблица категорий), Language Policy, Working Protocol, Global Pre-Flight.
- В каждой категории `docs/<category>/` **MUST** лежать собственный `AGENTS.md` с фиксированными секциями: Scope, When to Read, Working Protocol, Hard Invariants, Pre-Flight Checklist, «If Chapters Disagree».
- Корневой `AGENTS.md` **MUST NOT** содержать категорийных правил. Категорийный `AGENTS.md` **MUST NOT** содержать правил, отсутствующих в главах (он — навигация + выжимка инвариантов).
- Шаблон категорийного `AGENTS.md` закреплён в [`../_meta/templates/AGENTS-category.md`](../_meta/templates/AGENTS-category.md).
- При конфликте между правилом в главе и выжимкой в `AGENTS.md` — глава авторитетна. Это декларируется секцией «If Chapters Disagree» в каждом `AGENTS.md`.

---

## Consequences

### Positive

- Агенту гарантирован единый entry-point, независимо от задачи.
- Категории изолированы: добавление новой категории не требует правки корневого `AGENTS.md`, кроме одной строки в Document Map.
- Навигация и правила развиваются независимо.
- Шаблон `AGENTS.md` делает структуру предсказуемой — агент знает, где искать Hard Invariants и Pre-Flight.

### Negative

- Дублирование формулировок инвариантов между `AGENTS.md` и главами требует дисциплины (правка в главе — правка в `AGENTS.md`). Частично смягчается тем, что `AGENTS.md` содержит **ссылку** на главу, а не полную формулировку правила.
- Две уровня навигации — чуть больше чтения для простых задач (агент открывает корневой → категорийный → главу).

### Neutral

- Количество файлов растёт линейно с числом категорий: каждая категория добавляет минимум `AGENTS.md` и `README.md`.

---

## Alternatives considered

- **Один большой `CLAUDE.md` / `AGENTS.md` со всеми правилами.** Отклонено: контекст агента не резиновый; при росте правил становится невозможно загрузить целиком; правки превращаются в merge-конфликты; нет способа отключить часть правил для не-релевантных задач.
- **Плоская структура `docs/*.md` без категорий.** Отклонено: нет способа сгруппировать правила по домену; агент вынужден читать все заголовки, чтобы выбрать нужные; не масштабируется.
- **Один `AGENTS.md` + иерархия глав без промежуточных `AGENTS.md`.** Отклонено: корневой файл перегружается маршрутизацией; правки в категории тянут правки в корне; Hard Invariants каждой категории вынуждены смешиваться.
- **Машино-читаемый index (YAML/JSON) вместо markdown-`AGENTS.md`.** Отклонено: агенты LLM лучше работают с прозой; человек тоже редактирует документ; маркдаун читается и рендерится без специального парсера.

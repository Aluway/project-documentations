# ADR 0003: FSD 2.1 «pages-first» как архитектурная методология фронтенда

> Оглавление: [`README.md`](README.md).

---

## Status

accepted

---

## Context

Фронтенд-проекты на React+TS без явной архитектурной методологии со временем накапливают:

- хаотичную структуру папок (`components/`, `utils/`, `helpers/`, вложенные глубже трёх уровней);
- циклические зависимости между «модулями» без явного направления импортов;
- «божественные» Context'ы и stores, используемые из произвольных мест;
- невозможность переиспользовать блоки без рефакторинга на несколько тысяч строк.

Feature-Sliced Design (FSD) — методология, стандартизирующая структуру через три уровня: слои (`app`, `pages`, `widgets`, `features`, `entities`, `shared`), слайсы (по бизнес-домену), сегменты (по технической роли: `ui`, `api`, `model`, `lib`, `config`).

FSD существует в двух актуальных версиях:

- **v2.0** — entities-first: код по умолчанию декомпозируется в `entities/*` и `features/*`; страницы собирают готовое.
- **v2.1** — pages-first: код по умолчанию живёт на странице; в нижние слои выносится только при доказанном переиспользовании.

v2.0 создаёт множество пустых «на будущее» слайсов, которые никогда не переиспользуются, и приводит к преждевременным абстракциям. v2.1 решает это.

---

## Decision

Принято следующее:

- Архитектура фронтенд-кода в репозитории **MUST** следовать Feature-Sliced Design v2.1.
- Новый код **MUST** размещаться по **pages-first flow**: по умолчанию — в слайсе страницы (`pages/<page-name>/`). Вынос в `widgets/*`, `features/*`, `entities/*` — **только при доказанном переиспользовании** (≥ 2 потребителей) или чётком v2.1-критерии (например, виджет как крупная независимая область страницы).
- Направление импортов **MUST** быть строго сверху вниз: `app → pages → widgets → features → entities → shared`. Нижние слои **MUST NOT** импортировать из верхних.
- Слайсы одного слоя **MUST NOT** импортировать друг друга. Единственное исключение — `entities/*` через `@x`-фасады (`entities/<exporter>/@x/<consumer>.ts`).
- У каждого слайса **MUST** быть `index.ts` с явными именованными ре-экспортами. `export *` — **MUST NOT**.
- Сегменты **MUST** именоваться по назначению: `ui`, `api`, `model`, `lib`, `config`. Essence-имена (`components`, `hooks`, `types`, `utils`, `helpers`) — **MUST NOT**.
- Слой `processes` из FSD v1 — **deprecated** и **MUST NOT** появляться в новом коде.
- Правила детализированы в [`../architecture/`](../architecture/AGENTS.md), разбитой по главам 01-layers, 02-slices-segments, 03-public-api, 04-cross-imports, 05-pages-first, 06-import-rules, 07-checklists, 08-examples.

Существующий код, написанный под v2.0, остаётся валидным (v2.1 не ломает совместимость). Миграция — по запросу, не принудительно.

---

## Consequences

### Positive

- Единая структура, стандартизированная внешней методологией с официальной документацией — снижает bus-factor.
- pages-first снижает порог входа: новичок знает, куда класть код по умолчанию.
- Запрет преждевременных абстракций (`entities/`, `features/` «на будущее») уменьшает шум в кодовой базе.
- Направление импортов делает невозможными циклические зависимости между слоями.
- `@x`-фасады стандартизируют межсущностные ссылки, которые в v2.0 были неформальны.

### Negative

- Ограничение «слайсы одного слоя не импортируют друг друга» изредка приводит к неочевидным решениям (где должен жить общий код двух фич — вниз в entities? в shared? на родительской странице?).
- Для команд, знакомых только с CRA-структурой, — кривая обучения на 1–2 недели.
- pages-first иногда требует «обратной миграции»: код, помещённый на странице, позже обнаруживает второго потребителя и переезжает в нижний слой. Это стоит одного рефакторинга, но ловится чек-листом F в [`../architecture/07-checklists.md`](../architecture/07-checklists.md).

### Neutral

- Привязка к внешней методологии означает, что обновления FSD (v2.2, v3.0) требуют явного принятия — через новый ADR.

---

## Alternatives considered

- **Свободная структура без методологии.** Отклонено: см. Context — проблема, которую FSD решает.
- **FSD v2.0 (entities-first).** Отклонено: приводит к преждевременным абстракциям и пустым слайсам; v2.1 явно решает эти проблемы, оставаясь совместимой.
- **Clean Architecture / Hexagonal Architecture.** Отклонено: избыточна для фронтенда; оптимизирована под домены с тяжёлым business logic, а не под UI; в React-экосистеме нет стандартных инструментов поддержки.
- **Atomic Design (atoms / molecules / organisms / templates / pages).** Отклонено: оптимизирована под UI-kit, не под приложения; не регулирует бизнес-логику и данные; не даёт правил импорта; плохо сочетается со state-менеджерами.
- **Модульная структура по бизнес-доменам без слоёв** (Domain-Driven Design в чистом виде). Отклонено: не решает проблему технической декомпозиции (UI vs API vs state); команда и так знает свои домены, но не знает, где лежит UI и где API.

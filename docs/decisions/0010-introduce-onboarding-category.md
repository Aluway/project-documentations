# ADR 0010: Ввод категории `docs/onboarding/` для fork-bootstrap и первых шагов

> Оглавление: [`README.md`](README.md).

---

## Status

accepted

---

## Context

Шаблонный репозиторий уже содержит 6 содержательных категорий (architecture, code-style, workflow, security, decisions, _meta), корневой [`../../README.md`](../../README.md) с общим описанием и [`../../AGENTS.md`](../../AGENTS.md) как hub для агентов. Однако **первый день после форка** — сценарий «я только что склонировал шаблон, с чего начать?» — недостаточно покрыт:

- Корневой [`../../README.md`](../../README.md) содержит 5-пунктовую инструкцию «Using this template», но она беглая: «fill `PROFILE.md`, review decisions, run lint». Нет пошагового чек-листа по каждому полю `PROFILE.md`, по правке корневого README, по настройке `CODEOWNERS`, по выбору LICENSE.
- [`../_meta/agent-smoke-tests.md`](../_meta/agent-smoke-tests.md) — инструмент квартального ревью и верификации после структурных изменений; он не про первый день.
- Нет документа, описывающего типовые развилки форка: «мы монорепо», «у нас есть backend», «мы на Expo/Electron/SSR-first», «нам нужен i18n».
- Нет red-flag'ов: «если после форка X, Y, Z — что-то пошло не так».

Без такой точки входа форк проходит через 30-минутный bus-factor-зависимый опыт: кто-то из команды, знакомый с template'ом, должен провести новичка за руку. Это работает в небольших командах и ломается при росте / смене состава / одиночном форке.

Нужна отдельная категория `docs/onboarding/`, собирающая:

- Чек-лист первого дня (10–15 конкретных шагов).
- Разветвления под типовые сценарии проекта.
- Red-flag'и и «что делать, если…».
- Связь с остальными категориями как маршрут «читать в таком порядке».

---

## Decision

Принято следующее:

- **MUST** создать категорию [`../onboarding/`](../onboarding/AGENTS.md) с двумя главами:
  - `01-first-fork.md` — чек-лист первого дня после форка: от клонирования до первого коммита.
  - `02-common-branches.md` — разветвления: что делать, если проект — монорепо / fullstack / Expo / Next.js / без CI и т.п.
- Категория **MUST** следовать шаблону [`../_meta/templates/AGENTS-category.md`](../_meta/templates/AGENTS-category.md).
- Категория — **проект-агностична**. Разветвления описывают **общие паттерны**, не навязывают конкретные инструменты. Для специфичных проектов форк создаёт собственный onboarding-подглаву через override-ADR.
- Корневой [`../../README.md`](../../README.md) **MUST** ссылаться на `docs/onboarding/` как на «начните отсюда после форка».
- [`../../AGENTS.md`](../../AGENTS.md) Document Map **MUST** включать onboarding-категорию со специальной пометкой «start here for new forks».
- Главы onboarding **MUST** содержать явные проверяемые checkboxes, не абстрактные принципы. Цель — чтобы новичок с `docs/onboarding/01-first-fork.md` смог пройти путь за один рабочий день без вопросов к мейнтейнерам.

---

## Consequences

### Positive

- Первый день после форка становится детерминированным: есть чек-лист, есть red-flag'и, есть развилки.
- Снижается bus-factor-зависимость команды-мейнтейнера шаблона — форк не требует персонального туториала.
- Корневой README освобождается от детального tutorial-контента — остаётся обзорный. Детали — в категории.
- Smoke-tests остаются в `_meta/` как инструмент ревьюера/governance, onboarding остаётся для нового форка — разделение аудитории.

### Negative

- Седьмая категория — ещё один пункт в Document Map. Смягчается тем, что onboarding — категория первого визита, после него не возвращаются, пока не форкается следующий проект.
- Часть содержимого onboarding'а дублирует корневой README (шаг «заполнить PROFILE»). Дубликат — **намеренный**: README — landing для GitHub, onboarding — детальный чек-лист. Обе аудитории нужны.

### Neutral

- Категория не меняет существующие правила; она — мета-документация о процессе адопции template'а.

---

## Alternatives considered

- **Оставить всё в корневом README.** Отклонено: README уже ~70 строк; добавление детального чек-листа делает его слишком длинным для landing-page GitHub. Лучше короткий landing + глубокая категория.
- **Положить в `_meta/onboarding.md` как файл.** Отклонено: `_meta/` — про документацию о документации (шаблоны, стиль, governance). Onboarding — про адопцию; другая аудитория и scope.
- **Слить с `_meta/agent-smoke-tests.md`.** Отклонено: smoke-tests — инструмент квартального ревью (владелец: governance); onboarding — первый день нового форка (владелец: adoption). Разные use case'ы, разный жизненный цикл, разная каденция обновлений.
- **Category `docs/tutorials/`.** Отклонено: «tutorial» подразумевает поэтапное изучение; здесь нужен **чек-лист**, не учебник.

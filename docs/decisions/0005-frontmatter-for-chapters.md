# ADR 0005: Обязательный YAML-frontmatter в каждой главе документации

> Оглавление: [`README.md`](README.md).

---

## Status

accepted

---

## Context

Документация стареет молча: правила остаются в силе годами даже после того, как их контекст перестал быть актуальным. Без машино-читаемых метаданных:

- Нет способа определить, давно ли главу ревьюили.
- Нет способа связать variant-главу с её условием активации автоматически.
- Нет способа отличить `draft` от `active` от `deprecated` без чтения всего текста.
- Нет единого источника для семантического версионирования правил (breaking vs дополнение vs переформулировка).

Проверка CI, отвечающая на эти вопросы, невозможна без формальных метаданных в файле. Имя файла и содержимое заголовков — не формальные метаданные: они для людей.

YAML-frontmatter — устоявшийся формат (Jekyll, Hugo, Gatsby, Obsidian, Docusaurus). Парсится любым markdown-процессором; не мешает рендерингу; не конфликтует с GitHub-рендером (скрывается или рендерится как таблица в зависимости от версии).

---

## Decision

Принято следующее:

- Каждая глава в `docs/**/*.md` (включая `AGENTS.md`, `README.md`, `PROFILE.md`) **MUST** начинаться с YAML-frontmatter.
- Обязательные поля: `version`, `last-reviewed`, `status`. Формат — [`../_meta/frontmatter.md`](../_meta/frontmatter.md) раздел 2.
- Для variant-глав — дополнительно поле `requires` с секциями `profile` и `min`. Формат — [`../_meta/frontmatter.md`](../_meta/frontmatter.md) раздел 3.
- Для ADR — дополнительно поля `id`, `date`, `supersedes`, `superseded-by`. Формат — [`../_meta/frontmatter.md`](../_meta/frontmatter.md) раздел 4.
- `version` — semver, bump'ится по правилам: MAJOR — breaking правила, MINOR — новое правило, PATCH — формулировка/пример.
- `last-reviewed` — ISO-дата, **MUST** обновляться при любой правке **и** при плановом ревью (даже без правки содержимого).
- `status` — одно из: `active`, `draft`, `deprecated` (для глав) / `proposed`, `accepted`, `rejected`, `superseded`, `deprecated` (для ADR).
- Добавление новых полей во frontmatter **MUST** сопровождаться новым ADR и миграцией всех существующих файлов в том же PR.
- Старые файлы без frontmatter мигрируются однократным PR. После миграции — коммит без frontmatter в `docs/**/*.md` **MUST NOT** приниматься в мёрж.

---

## Consequences

### Positive

- Машино-читаемая актуальность: агент видит `last-reviewed` и решает, можно ли опираться на главу без дополнительной проверки.
- Машино-проверяемая активация variant-глав: `requires.profile` сверяется с `PROFILE.md` автоматически.
- Semver-версионирование правил: breaking-правки видны через bump MAJOR, что сигнализирует авторам зависящих репо о необходимости проверить свой код.
- Готовая основа для CI-линтеров: валидация YAML, валидация полей, проверка ссылочной целостности `supersedes` в ADR.
- Формальный жизненный цикл главы: `draft` → `active` → `deprecated` — через изменение `status`, а не через неформальные пометки в тексте.

### Negative

- Дополнительная бюрократия при создании нового файла — нужно заполнить frontmatter.
- Ответственность за `last-reviewed` и `version` — на авторе правки; без CI-линтера дисциплина держится на ревью.
- Начальная миграция существующих файлов — однократная разовая работа, но трогает каждую главу.

### Neutral

- GitHub-рендер отображает YAML-frontmatter как таблицу в начале файла. Для некоторых читателей это полезно (видны метаданные), для других — шум. Соглашение — не пытаться скрывать.

---

## Alternatives considered

- **Без формальных метаданных.** Отклонено: см. Context — проблема, которую frontmatter решает.
- **Метаданные в сопутствующем `.yaml`-файле рядом с главой.** Отклонено: разрыв между файлом правила и его метаданными; правки не атомарны; вдвое больше файлов в репо.
- **Метаданные в первом абзаце markdown («Версия 1.0, последний ревью 2026-04-17»).** Отклонено: не машино-читаемо без кастомного парсера; легко расходится с реальностью.
- **JSON-frontmatter вместо YAML.** Отклонено: менее читаем для человека; не поддерживается частью markdown-инструментов; YAML — устоявшийся стандарт во frontmatter-экосистеме.
- **Использовать только git-историю для датировки.** Отклонено: git-commit фиксирует «когда правили», но не «когда ревьюили без правки»; содержательная актуальность ≠ время последнего `git touch`.

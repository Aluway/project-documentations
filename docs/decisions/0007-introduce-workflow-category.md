# ADR 0007: Ввод категории `docs/workflow/` для git-процесса, PR и релизов

> Оглавление: [`README.md`](README.md).

---

## Status

accepted

---

## Context

Процессные правила — git-конвенция, PR-шаблон, code-review, релизы — на момент принятия этого ADR были распределены по нескольким местам:

- Корневой [`../../AGENTS.md`](../../AGENTS.md) — «Commit message follows `<type>(<scope>): <subject>` convention (see repo conventions)» в Global Pre-Flight, без самих правил.
- Глобальный `CLAUDE.md` пользователя (вне репо) — полная конвенция коммитов, типы, длина subject'а. Эти правила **не переносимы**: при форке репо в другой контекст они не инкорпорируются.
- Архитектурные чеклисты в [`../architecture/07-checklists.md`](../architecture/07-checklists.md) — упоминают git-операции (например, «локально проходят тайпчек и тесты» перед коммитом), но не описывают их.

Это создаёт три проблемы:

1. **Не переносимо.** Форк репо-шаблона не наследует git-правила, которые живут в личном глобальном конфиге автора.
2. **Не обнаруживаемо.** Агент, читая корневой `AGENTS.md`, видит ссылку «see repo conventions» и не знает, куда смотреть.
3. **Не версионируемо.** Правила вне репо не проходят через PR, не имеют `last-reviewed`, не линтуются.

Решение — собрать процессные правила в отдельную категорию `docs/workflow/` по тому же паттерну, что `architecture/` и `code-style/`. Категория стек-агностична и процесс-ориентирована.

---

## Decision

Принято следующее:

- **MUST** создать категорию [`../workflow/`](../workflow/AGENTS.md) с четырьмя главами:
  - `01-git.md` — ветвление, конвенция коммитов, атомарность, работа с публичной историей, политика lockfile'ов и `.gitignore`.
  - `02-pull-requests.md` — размер PR, заголовок, шаблон описания, draft-режим, связка с issues.
  - `03-code-review.md` — кто ревьюит, SLA, чек-лист ревьюера, как давать и принимать feedback.
  - `04-releases.md` — semver, changelog, tagging, hotfix-процесс.
- Категория **MUST** следовать шаблону [`../_meta/templates/AGENTS-category.md`](../_meta/templates/AGENTS-category.md): Scope, When to Read, Working Protocol, Hard Invariants, Pre-Flight Checklist, «If Chapters Disagree».
- Корневой [`../../AGENTS.md`](../../AGENTS.md) **MUST** делегировать git-правила новой категории (в Document Map и Global Pre-Flight).
- Правила категории **MUST** быть проект-агностичны — подходить шаблонному репо, а не конкретному продукту. Проект-специфичные отклонения (например, trunk-based vs git-flow) документируются в форке через override-ADR.
- Категория **MAY** дополнительно переопределить владение через `CODEOWNERS` в проекте-потомке; в template-репо владельца нет.

---

## Consequences

### Positive

- Процессные правила становятся частью репозитория: переживают форк, проходят через PR, версионируются, линтуются.
- Агент получает единую точку входа для git-задач вместо ссылки «see repo conventions».
- Корневой `AGENTS.md` освобождается от частных правил commit-format — уступает их категории, остаётся навигационным.
- Форкнутый проект может override-нуть главу через свой ADR (например, «в этом проекте trunk-based release model вместо tag-based») — стандартный механизм замены решения.

### Negative

- Добавляется пятая категория — небольшой рост когнитивной нагрузки при знакомстве с репо. Смягчается тем, что categori AGENTS.md имеют фиксированный шаблон.
- Часть правил дублирует то, что многие команды держат «в голове». Документирование очевидного кажется избыточным, но именно это делает стандарт переиспользуемым.

### Neutral

- Правила из этой категории не влияют на код напрямую — только на процесс. Pre-flight категории — короткий, проверяется перед коммитом.

---

## Alternatives considered

- **Оставить правила в корневом `AGENTS.md`.** Отклонено: корневой файл — навигация, не правила ([ADR 0001](0001-agents-hub-navigation.md)). Подробная commit-convention перегружает hub и нарушает принцип.
- **Держать в глобальном `CLAUDE.md` (как сейчас).** Отклонено: см. Context — не переносимо, не обнаруживаемо, не версионируется.
- **Свести в одну главу в `architecture/`.** Отклонено: процесс разработки — не архитектура; смешение размывает scope обеих категорий.
- **Создать `docs/process/` вместо `docs/workflow/`.** Отклонено: «workflow» точнее описывает содержание (git-флоу, PR-флоу, release-флоу); «process» слишком общо и может конфликтовать с продуктовыми процессами.
- **Разделить на две категории: `docs/git/` и `docs/release/`.** Отклонено: PR и code-review не ложатся ни туда, ни сюда естественно; четыре главы в одной категории — нормальный объём.

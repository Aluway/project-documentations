# 03 — Manual Memoization (variant)

> Оглавление: [`../README.md`](../README.md). Профиль стека: [`../PROFILE.md`](../PROFILE.md).
> **Активен, если в `PROFILE.md` React Compiler = disabled или N/A** (например, React 18 без компилятора).

Правила ручной мемоизации `useMemo` / `useCallback` / `React.memo`, когда компилятор не работает за вас.

---

## 1. Главный принцип

> **Не мемоизируйте преждевременно. Мемоизируйте осознанно.**

Мемоизация — не «сделать быстрее», а «избежать конкретного ре-рендера в конкретном месте». Добавлять её **MUST** только после:

1. Профилирования (React DevTools Profiler → видно избыточный ре-рендер).
2. Гипотезы: «этот ре-рендер не нужен, потому что X».
3. Подтверждения выигрыша: после мемо — ре-рендер исчезает.

---

## 2. `useMemo`

Применяется для:
- **Дорогого вычисления** (> 1 мс) в теле компонента.
- **Стабильной ссылки** на объект/массив, передающийся в мемоизированный дочерний компонент или как зависимость `useEffect`.

```tsx
const filtered = useMemo(
  () => items.filter((item) => item.active),
  [items],
);
```

- Не применяйте для примитивов (`const total = useMemo(() => a + b, [a, b])`) — оверхед мемо больше самого вычисления.
- Не применяйте без измерения — профиль должен показать улучшение.

---

## 3. `useCallback`

Применяется для **стабильной ссылки на функцию**, когда:
- Функция передаётся как prop в мемоизированный (`React.memo`) дочерний компонент.
- Функция — зависимость `useEffect`.
- Функция передаётся в сторонние либы, требующие референсного равенства.

```tsx
const handleSubmit = useCallback(
  (data: FormData) => { submit(data); },
  [submit],
);
```

- Если дочерний компонент не `React.memo` и функция не в зависимости эффекта — `useCallback` **SHOULD NOT** использоваться: нет выигрыша.

---

## 4. `React.memo`

Оборачивает компонент, чтобы он не ре-рендерился, если props референсно равны.

```tsx
export const UserCard = React.memo(function UserCard({ user }: UserCardProps) {
  return <article>{user.name}</article>;
});
```

Применяется для:
- **Тяжёлых компонентов**, часто ре-рендерящихся с одними и теми же props.
- **Списков**, где большинство элементов не меняется при обновлении массива.

Не применяется, если:
- Props почти всегда разные при ре-рендере (мемо не помогает).
- Props содержат non-primitive значения без мемоизации в родителе (мемо не сработает).

### Кастомный comparator

```tsx
export const UserCard = React.memo(UserCardImpl, (prev, next) => prev.user.id === next.user.id);
```

- **SHOULD NOT** использовать, если хватает default-поверхностного сравнения. Кастомный comparator — источник багов при эволюции props.

---

## 5. Правило «снизу вверх»

Мемоизацию **SHOULD** ставить снизу вверх:

1. Сначала мемо тяжёлого листа (`React.memo`).
2. Затем стабилизировать props, передаваемые в него (`useMemo`/`useCallback` в родителе).
3. Проверить в profiler, что ре-рендер устранён.

Обратный порядок (родитель сначала) часто не даёт эффекта.

---

## 6. Когда мемоизация **не** нужна

- Компонент ре-рендерится, но рендер быстрый (< 1 мс). Оверхед мемо > выигрыш.
- Компонент — ближе к корню дерева и редко ре-рендерится сам по себе.
- Props состоят только из примитивов и стабильны по природе.

---

## 7. Альтернатива мемоизации — декомпозиция

Перед добавлением `memo` / `useMemo` / `useCallback` **SHOULD** рассмотреть переструктурирование дерева как более дешёвую альтернативу:

- Вынести часто обновляемый state в меньший компонент.
- Переместить провайдер ближе к потребителям.
- Переделать на uncontrolled / ref, если контролируемое решение порождает каскад ре-рендеров.

---

## 8. Антипаттерны

- **Профилактическая мемоизация всего подряд** — **MUST NOT**. Замедляет приложение, усложняет код.
- **`useCallback` без мемоизированного потребителя** — **SHOULD NOT**: нет выигрыша.
- **`useMemo` на примитив** — **SHOULD NOT**.
- **Забытая / неполная зависимость в `useMemo`/`useCallback`** — **MUST NOT**: `react-hooks/exhaustive-deps` должен быть `error`.
- **Мемо + мутация props/state** — **MUST NOT**: мутация ломает мемо.

# 04 — Модель состояния (universal)

> Оглавление: [`../README.md`](../README.md). Профиль стека: [`../PROFILE.md`](../PROFILE.md).

Универсальная ментальная модель распределения состояния по слоям. Конкретные инструменты (TanStack Query, Zustand, Redux Toolkit и др.) — в соответствующих `variants/state-*.md`.

---

## 1. Пять слоёв состояния

| Слой | Что хранит | Как решать |
|---|---|---|
| **1. Server state** | Данные, источник истины которых — бэкенд: списки, сущности, пагинация, кеш | Специализированный менеджер серверного state (см. `variants/state-*.md` под свой инструмент) |
| **2. Global client state** | UI-состояние, не зависящее от сервера: тема, локаль, сессия, открытость сайдбара | Zustand / Redux Toolkit / Jotai / Context (см. `variants/state-*.md`) |
| **3. Local component state** | Флаги, значения инпутов, открыт/закрыт | `useState` / `useReducer` |
| **4. Form state** | Поля формы, валидация, submit-флаги | Инструмент согласно `PROFILE.md`: React 19 Actions / React Hook Form / native |
| **5. URL state** | Состояние, которое должно быть shareable/bookmarkable: фильтры, пагинация, выбранная вкладка | Search params через роутер |

**Главное правило:** любое новое состояние **MUST** быть отнесено к одному из слоёв. Смешивание слоёв — антипаттерн.

---

## 2. Решающее дерево

1. **Это серверные данные?** → слой 1, специализированный менеджер.
2. **Это состояние, которое нужно shareable через URL?** → слой 5, search params.
3. **Это state нужен более чем одной соседней ветке дерева?** → слой 2 (глобальный клиентский) или поднять в общий родитель.
4. **Это поля формы?** → слой 4, инструмент форм из `PROFILE.md`.
5. **Иначе** → слой 3, локальный `useState`/`useReducer`.

---

## 3. Как слои взаимодействуют

```
┌─────────────────────────────────────────┐
│  URL state (5)  — источник для фильтров │
└─────────────────────────────────────────┘
              │ читает
              ▼
┌─────────────────────────────────────────┐
│  Server state (1)  — параметризуется    │
│  фильтрами из URL                       │
└─────────────────────────────────────────┘
              │ читает
              ▼
┌─────────────────────────────────────────┐
│  Form state (4)  — инициализируется     │
│  из Server state при редактировании      │
└─────────────────────────────────────────┘
              │ пишет
              ▼
┌─────────────────────────────────────────┐
│  Server state (1)  — получает мутацию    │
└─────────────────────────────────────────┘

Parallel:
┌─────────────────────────────────────────┐
│  Global client state (2) — UI-часто     │
│  ре-рендерящие переключатели            │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│  Local state (3) — внутри одного       │
│  компонента и ниже                      │
└─────────────────────────────────────────┘
```

---

## 4. Расширение модели

Если у команды есть специфичные слои (например, собственная feature-flags система, observable storage) — вынесите в отдельный модуль в `variants/` и опишите правила для него. Не мешайте с базовыми пятью слоями.

---

## 5. Антипаттерны

- **Глобальный Context для серверных данных** — **MUST NOT**. Ваш менеджер серверного state уже глобально-доступный кеш.
- **Зеркалирование server state в client store** — **MUST NOT**. Источник истины один.
- **`useEffect` для синхронизации props → state** — **MUST NOT** (см. [`03-hooks.md`](03-hooks.md) раздел 4).
- **Хранение URL-состояния в `useState` без синхронизации с URL** — **SHOULD NOT**. Поломает refresh и share.
- **Локальный state, который по сути должен быть в URL** (текущий tab, фильтры таблицы) — **SHOULD NOT**.
- **Смешивание слоёв в одном хранилище** (UI-состояние и серверные данные в одном Redux-срезе без разделения) — **SHOULD NOT**. Разделяйте по слою.

# AGENTS.md — Architecture (FSD 2.1)

> Категорийная точка входа для ИИ-агентов.
> Правила FSD 2.1 авторитетны для всего фронтенд-кода в этом репозитории. Главы написаны на русском.

---

## Scope

Эта категория покрывает **только архитектурные правила фронтенда по методологии Feature-Sliced Design v2.1 («pages-first»)**:

- Что куда класть (слои, слайсы, сегменты).
- Как описывать Public API слайса.
- Как делать кросс-ссылки между сущностями (`@x`).
- Правила импортов.

Всё остальное (стиль кода, React-паттерны, тестирование, стили) — вне scope'а; см. [`../code-style/AGENTS.md`](../code-style/AGENTS.md).

## Working Protocol

Для любого изменения фронтенд-кода:

1. **Ориентация** — если задача в незнакомой области, пробегите [`01-layers.md`](01-layers.md), чтобы подтвердить ответственность слоя.
2. **Размещение** — прогоните дерево решений в [`05-pages-first.md`](05-pages-first.md). По умолчанию — слайс страницы; ниже двигаем только при доказанном переиспользовании.
3. **Правила слайса** — [`02-slices-segments.md`](02-slices-segments.md) для именования, [`03-public-api.md`](03-public-api.md) для контракта `index.ts`.
4. **Импорты** — соблюдайте [`06-import-rules.md`](06-import-rules.md): направление слоёв, алиасы, `import type`.
5. **Кросс-entity ссылки** — если двум `entities/*` нужен общий тип, используйте `@x`-фасады по [`04-cross-imports.md`](04-cross-imports.md). Кросс-импорты на других слоях запрещены.
6. **Верификация перед финишем** — прогоните подходящий чеклист из [`07-checklists.md`](07-checklists.md) плюс [глобальный pre-flight](07-checklists.md#i-архитектурный-pre-flight-на-каждый-коммит). Устраните все несоответствия.

## Hard Invariants

Проверьте перед завершением любой задачи, затрагивающей структуру кода. Полные правила — в главах выше; этот список — быстрые ворота.

- Направление слоёв: `app → pages → widgets → features → entities → shared`. Нижние слои **MUST NOT** импортировать из верхних.
- Слайсы одного слоя **MUST NOT** импортировать друг друга. Единственное исключение: `entities/*` через `@x`-фасады (файл на конкретного потребителя: `entities/<exporter>/@x/<consumer>.ts`).
- У каждого слайса есть `index.ts` в корне с явными именованными ре-экспортами. Никаких `export *`.
- Внешние потребители импортируют слайсы (`pages`, `widgets`, `features`, `entities`) через `index.ts` — никогда не через внутренние пути. У `app` и `shared` нет слайсов — импорт из сегмента/файла напрямую.
- Сегменты именуются по назначению (`ui`, `api`, `model`, `lib`, `config`). Essence-имена (`components`, `hooks`, `types`, `utils`, `helpers`) запрещены.
- Pages-first: код по умолчанию живёт на странице использования. Пустые «на будущее» `entities/*` и `features/*` **MUST NOT** создаваться.

## Pre-Flight Checklist

Короткий прогон перед завершением задачи. Полные чеклисты по типам задач — в [`07-checklists.md`](07-checklists.md).

- [ ] Код размещён по pages-first flow (см. [`05-pages-first.md`](05-pages-first.md)).
- [ ] Направление слоёв соблюдено: `app → pages → widgets → features → entities → shared`.
- [ ] Слайсы одного слоя не импортируют друг друга (исключение: `entities/*` через `@x`).
- [ ] У каждого нового слайса есть `index.ts` с явными именованными ре-экспортами (никаких `export *`).
- [ ] Потребители импортируют через корень слайса, не через внутренние пути.
- [ ] Имена сегментов из стандартного набора (`ui`, `api`, `model`, `lib`, `config`).
- [ ] Нет пустых «на будущее» `entities/*` и `features/*`.

Глубокая проверка — [глобальный pre-flight (раздел I)](07-checklists.md#i-архитектурный-pre-flight-на-каждый-коммит).

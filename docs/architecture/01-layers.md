# 01 — Слои

> Оглавление: [`README.md`](README.md).

Слои — верхний уровень организации. Модуль **MUST** импортировать только из слоёв строго **ниже** своего.

```
app      ← верх
pages
widgets
features
entities
shared   ← низ
```

> **Историческая заметка — слой `processes`.**
> В ранних ревизиях FSD между `app` и `pages` был слой `processes` для сложных сквозных сценариев. В v2.1 он **deprecated** и **MUST NOT** появляться в новом коде. Существующий код, который его использует, при ближайшем рефакторинге **SHOULD** быть перенесён в `pages` или `app`.

---

## `app`

**Назначение:** всё, что заставляет приложение работать.

**Содержит только сегменты — без слайсов.**

- Конфигурация роутинга (композиция маршрутов — см. [`09-routing.md`](09-routing.md))
- Точка входа приложения
- Глобальные провайдеры (store, тема, i18n, query-клиент)
- Глобальные стили
- Инициализация аналитики для всего приложения
- DI-wiring для `shared/*`, которому нужны runtime-значения из домена (см. [`10-migration-guide.md` §11](10-migration-guide.md#11-инверсия-shared--upper-через-di-в-app))

**MUST:**
- Содержать сегменты напрямую, без слоя слайсов.
- Быть единственным местом, где собирается приложение целиком.

**MUST NOT:**
- Содержать бизнес-логику, которая могла бы жить в `pages/*` или ниже.

✓ Пример раскладки:
```
app/
├── providers/
├── routes/
├── styles/
└── index.ts
```

---

## `pages`

**Назначение:** экраны (маршруты) приложения.

**Содержит слайсы.** Один слайс на страницу или один слайс на группу тесно связанных страниц (`sign-in` + `sign-up`).

- UI-раскладка страницы
- Состояния загрузки / ошибки / пустого вида
- Локальные для страницы запросы данных
- Код, который не переиспользуется вне страницы — **остаётся здесь** (pages-first в v2.1)

**MUST:**
- Экспонировать Public API через `index.ts`.
- Хранить локальный UI, состояние и логику страницы здесь, пока переиспользование не подтверждено.

**MUST NOT:**
- Импортировать из соседних `pages/*`.
- Раскрывать внутренности страницы нижним слоям.

✓ Пример раскладки:
```
pages/
└── dashboard/
    ├── index.ts
    ├── ui/
    │   ├── dashboard-page.tsx
    │   └── revenue-chart.tsx    ← локален для страницы, не извлекаем
    ├── model/
    │   └── use-dashboard-data.ts
    └── api/
        └── get-dashboard.ts
```

## `widgets`

**Назначение:** самодостаточные UI-блоки, реализующие целостный use case, переиспользуемые между страницами **или** крупные независимые области одной страницы, которым полезна изоляция.

**Изменение v2.1:** виджет **MAY** владеть собственным состоянием, API и бизнес-логикой. Он больше не ограничен ролью чистой композиции. Практический маркер: если у блока есть `useState` / `useEffect` на локальные UI-настройки (collapse/expand, open/close, persist в `localStorage`) — это нормальный v2.1-виджет, а не «слишком много логики».

**MUST:**
- Экспонировать Public API через `index.ts`.
- Либо переиспользоваться на ≥ 2 страницах, либо быть достаточно крупной независимой областью, чтобы оправдать изоляцию в рамках одной страницы.

**MUST NOT:**
- Создаваться для мелких одностраничных блоков, которые комфортно уживаются внутри слайса страницы.

**Диагностика widget vs app** (типовая путаница для layout-файлов — `Sidebar`, `BottomNav`, `Layout`):

- **Если удалить файл, приложение без него не соберётся?** → `app/` (композиция, не блок). Пример: `Layout.tsx`, который расставляет виджеты и page-slot — живёт в `app/`.
- **Если удалить файл, страница откроется в «голом» режиме, но рендер не сломается?** → `widgets/`. Пример: `Sidebar` с собственным `useState(collapsed)` — живёт в `widgets/`.

Та же логика — для `ProtectedRoute` и прочих guard'ов: это **app-layer**, не widget (см. [09-routing.md](09-routing.md)).

✓ Пример раскладки:
```
widgets/
└── header/
    ├── index.ts
    ├── ui/
    │   └── header.tsx
    └── model/
        └── use-header-session.ts
```

## `features`

**Назначение:** переиспользуемые пользовательские взаимодействия с бизнес-ценностью (например, «добавить в корзину», «лайкнуть пост», «переключить подписку»).

**MUST:**
- Представлять взаимодействие, переиспользуемое на ≥ 2 страницах или виджетах.
- Экспонировать Public API через `index.ts`.

**MUST NOT:**
- Принимать любое мелкое взаимодействие. В слой попадают только переиспользуемые бизнес-значимые.

✓ Пример раскладки:
```
features/
└── add-to-cart/
    ├── index.ts
    ├── ui/
    │   └── add-to-cart-button.tsx
    ├── api/
    │   └── add-to-cart.ts
    └── model/
        └── use-add-to-cart.ts
```

---

## `entities`

**Назначение:** доменные модели — устойчивые бизнес-сущности: `user`, `product`, `order`.

**MUST:**
- Экспонировать Public API через `index.ts`.
- Для любых кросс-ссылок между сущностями использовать нотацию `@x`. См. [04-cross-imports.md](04-cross-imports.md).

**MUST NOT:**
- Импортировать другие entity-слайсы через их обычный Public API.
- Зависеть от `features/*`, `widgets/*`, `pages/*`, `app/*`.

✓ Пример раскладки:
```
entities/
├── user/
│   ├── index.ts
│   ├── ui/
│   ├── api/
│   ├── model/
│   └── @x/
│       └── post.ts      ← фасад для entities/post
└── post/
    ├── index.ts
    └── ...
```

## `shared`

**Назначение:** проект-независимый фундамент — то, что могло бы принадлежать любому проекту.

**Содержит только сегменты — без слайсов.**

- UI-кит (`shared/ui`)
- API-клиент и низкоуровневый транспорт (`shared/api`)
- Хелперы и библиотеки (`shared/lib`, организовано в подпапки по назначению)
- Константы и конфиги приложения (`shared/config`)
- Константы маршрутов — в `shared/config/routes`
- Инициализация i18n

**MUST:**
- Содержать только код без специфичных бизнес-знаний проекта.
- Импортироваться из любого слоя выше.

**MUST NOT:**
- Иметь структуру слайсов.
- Ссылаться на слои выше.

Примечание: приложение-осведомлённые элементы вроде констант маршрутов, логотипа, базовых URL API **допускаются** в `shared`, хоть они и «знают» о приложении — это уточнили в v2.1.

✓ Пример раскладки:
```
shared/
├── ui/
│   ├── button/
│   └── input/
├── api/
│   └── http-client.ts
├── lib/
│   ├── dates/
│   └── collections/
└── config/
    └── env.ts
```

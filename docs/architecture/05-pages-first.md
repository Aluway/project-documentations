# 05 — Pages-first: дерево решений (v2.1)

> Оглавление: [`README.md`](README.md).

Это самый важный раздел для ежедневного размещения кода. Когда вы сомневаетесь, куда положить новое — идите по этому дереву.

---

## Главный принцип

> **По умолчанию код живёт в слайсе страницы.**
> **Переносите в нижний слой только когда появляется конкретный случай переиспользования.**

Это переворачивает v2.0-менталитет, где сначала выделяли entities и features. В v2.1 вы **начинаете со страницы** и извлекаете вниз только при доказанном переиспользовании.

Почему:
- Меньше когнитивной нагрузки — большинство разработчиков и так думает страницами.
- Меньше преждевременных абстракций — пустые entities и features «на будущее» не создаются.
- Выше сплочённость кода — связанное лежит вместе, пока реально не потребуется разделить.

---

## Дерево решений

Отвечайте на вопросы по порядку. Останавливайтесь на первом совпадении.

### Шаг 1 — используется только одной страницей?

> Этот код используется **только** одной страницей, и переиспользовать его не планируется?

→ **MUST** остаётся в `pages/<page>/`. Внутри слайса делите по сегментам (`ui`, `model`, `api`, `lib`, `config`). Внутри `ui` при крупной странице — организуйте подпапками.

✓ Пример:
```
pages/dashboard/
├── index.ts
├── ui/
│   ├── dashboard-page.tsx
│   ├── revenue-chart.tsx      ← локален для страницы, не переиспользуется
│   └── activity-feed.tsx      ← локален для страницы, не переиспользуется
├── model/
│   └── use-dashboard-data.ts
└── api/
    └── get-dashboard.ts
```

### Шаг 2 — крупная независимая область одной страницы?

> Это **крупная, самодостаточная область** одной страницы, которой полезна отдельная граница для ясности, даже без переиспользования?

→ **MAY** стать слайсом `widgets/<name>/`. Это тот самый v2.1-кейс, когда виджет помогает структурировать большую страницу, не требуя переиспользования.

✓ Пример:
```
widgets/checkout-summary/     ← крупная область только страницы checkout
├── index.ts
├── ui/
├── model/
└── api/
```

**Не** извлекайте мелкий блок (одиночный график, сайдбар-карточка) только потому, что он «отделимый». Порог — оправдывают ли сложность и внутреннее состояние блока отдельный модуль.

### Шаг 3 — переиспользуется на ≥ 2 страницах?

> Код потребляется **двумя и более** страницами?

→ Извлекайте в `widgets/*` или `features/*`:

- **UI-ориентированный** блок (раскладка, составная секция интерфейса): `widgets/<name>/`.
- **Взаимодействие** с чёткой бизнес-ценностью (добавить в корзину, лайк, подписка, голос): `features/<name>/`.

✓ Пример виджета:
```
widgets/site-header/
└── ...
```

✓ Пример фичи:
```
features/like-post/
└── ...
```

### Шаг 4 — постоянный бизнес-объект?

> Код моделирует **доменный концепт**, встречающийся во многих фичах и виджетах (user, product, order, invoice, comment)?

→ **MUST** живёт в `entities/<name>/`. Слайс содержит доменные типы, доступ к данным, display-примитивы (например, `UserCard`), и store этой сущности.

Кросс-ссылки между сущностями — через `@x`. См. [04-cross-imports.md](04-cross-imports.md).

### Шаг 5 — проект-независим?

> Код **не привязан к бизнес-домену** — Button, fetch-обёртка, форматтер дат, общий хук?

→ **MUST** живёт в `shared/<segment>/`.

✓ Пример:
```
shared/ui/button/
shared/lib/dates/
shared/api/http-client.ts
```

## Анти-паттерны

### ✗ Создавать пустые entities «на будущее»

```
entities/user/
└── index.ts        ← пусто, реального кода ещё нет
```

**MUST NOT.** Ждите, пока появится реальный доменный код с реальным использованием. Преждевременная экстракция даёт churn: папки переименовывают, переносят или удаляют, когда реальные требования приходят. Периодическая проверка — [12-audit.md Check 8](12-audit.md#2-что-покрывает-audit).

### ✗ Преждевременно извлекать виджет

Перенос локального чарта из страницы в `widgets/revenue-chart/`, когда никакая другая страница его не использует, добавляет навигационных издержек с нулевой выгодой от переиспользования.

### ✗ Считать виджеты только композицией

v2.0-мышление: «виджеты только располагают features и entities, своей логики не держат». В v2.1 это **устарело**. Виджет **MAY** владеть своим store, API-вызовами и внутренней логикой — особенно для самодостаточной области страницы.

### ✗ Копипасты между страницами вместо извлечения

Если две страницы содержат одни и те же 50 строк — не оставляйте их дублированными. Случай переиспользования доказан — извлекайте в `widgets/*`, `features/*`, `entities/*` или `shared/*` по дереву.

## Паттерн миграции вниз (lift-down)

Когда изначально локальное для страницы нужно и в другом месте:

1. Определите тип слайса по шагам 3–5.
2. Создайте новый слайс с `index.ts`.
3. Перенесите код, сохранив структуру по сегментам.
4. Замените старый локальный импорт на импорт из Public API нового слайса.
5. Удалите копию в странице.

✓ До:
```
pages/dashboard/ui/activity-feed.tsx
pages/profile/ui/activity-feed.tsx      ← дубль
```

✓ После:
```
widgets/activity-feed/
├── index.ts
└── ui/
    └── activity-feed.tsx

pages/dashboard/ui/dashboard-page.tsx   → импортирует widgets/activity-feed
pages/profile/ui/profile-page.tsx       → импортирует widgets/activity-feed
```

## Шпаргалка

| Ситуация                                                 | Размещение                    |
|----------------------------------------------------------|-------------------------------|
| Используется одной страницей, компактное                 | `pages/<page>/`               |
| Используется одной страницей, крупная независимая область | `widgets/<name>/` (или страница) |
| UI-блок переиспользуется на ≥ 2 страницах                | `widgets/<name>/`             |
| Взаимодействие переиспользуется на ≥ 2 страницах          | `features/<name>/`            |
| Доменная модель, встречается во многих местах            | `entities/<name>/`            |
| Проект-независимое (UI-кит, утилиты, конфиг)             | `shared/<segment>/`           |

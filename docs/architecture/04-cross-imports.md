# 04 — Кросс-импорты (нотация `@x`)

> Оглавление: [`README.md`](README.md).

В FSD 2.1 нотация `@x` — это **стандартизированный Public API для кросс-импортов** между слайсами одного слоя. До v2.1 она была экспериментом, теперь — официальная часть методологии.

---

## Какую проблему решает

Слайсы одного слоя **MUST NOT** импортировать друг друга. Но некоторые доменные сущности законно связаны:

- у `post` есть `author: User`;
- в `order` лежат элементы `Product`;
- `comment` относится к `post` и `user`.

Без стандартизированного механизма разработчики либо дублируют типы, либо импортируют между слайсами в обход изоляции, либо придумывают самодельные костыли.

---

## Правило

`@x` **MUST** использоваться только на слое `entities/*`. На других слоях нужда в кросс-импорте — сигнал **пересмотреть декомпозицию**, а не повод применить `@x`.

---

## Механика

**Экспортёр** создаёт файл-фасад `@x/<consumer>.ts`, ре-экспортирующий ровно то, что разрешено видеть конкретному потребителю.

```
entities/
├── user/
│   ├── index.ts            ← Public API для остального приложения
│   ├── model/
│   │   └── types.ts
│   └── @x/
│       └── post.ts         ← фасад только для entities/post
└── post/
    ├── index.ts
    └── model/
        └── types.ts        ← импортирует User из @x-фасада
```

✓ Фасад экспортёра:
```ts
// entities/user/@x/post.ts
export type { User } from "../model/types";
```

✓ Потребитель:
```ts
// entities/post/model/types.ts
import type { User } from "@/entities/user/@x/post";

export interface Post {
  id: string;
  title: string;
  author: User;
}
```

✗ Без `@x` (запрещено):
```ts
// entities/post/model/types.ts
import type { User } from "@/entities/user";   // ✗ кросс-импорт одного слоя
```

---

## Зачем фасад на каждого потребителя?

Каждый `@x/<consumer>.ts` — **адаптированный** Public API: только те символы, которые нужны именно этому потребителю. Разные потребители могут видеть разные подмножества, это:

- Держит связанность узкой и явной.
- Делает очевидным, кто и от чего зависит (grep по `@x/post` даёт всех потребителей именем `post`).
- Позволяет экспортёру рефакторить внутренности без ломки несвязанных потребителей.

---

## Какие символы допустимы

Через `@x` **SHOULD** ре-экспортировать:

- Доменные типы (`type`, `interface`).
- Небольшие read-only селекторы/хелперы, чья форма является частью доменной модели.

Через `@x` **SHOULD NOT** ре-экспортировать:

- UI-компоненты (предпочтите композицию на верхних слоях).
- Состояния (store) — другой слайс не должен напрямую мутировать ваш store.
- Общие утилиты — им место в `shared/lib`.

---

## `@x` — последнее средство

Перед добавлением нового `@x`-фасада рассмотрите альтернативы по порядку:

1. Может общий кусок жить в `shared/`? Если он фреймворк-/проект-независим — да.
2. Может потребителю просто не нужно знать о связанной сущности? Иногда фича или виджет могут комбинировать две сущности на более высоком слое вместо того, чтобы одна entity импортировала другую.
3. Может две сущности слить в одну? Если они всегда используются вместе и делят идентичность — возможно, это один концепт изначально.

Только если ничего не подходит — добавляйте `@x`-фасад.

---

## Не-entity слои

Для `features/*`, `widgets/*`, `pages/*` правило строже:

> Если кажется, что на этих слоях нужен кросс-импорт — **MUST** пересмотреть декомпозицию.

Варианты:
- Сдвинуть общий кусок на слой ниже.
- Скомпозировать два слайса на более высоком слое, который уже видит оба (обычно страница или виджет).
- Вынести общую логику в `shared/lib`.

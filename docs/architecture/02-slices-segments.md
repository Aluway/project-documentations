# 02 — Слайсы и сегменты

> Оглавление: [`README.md`](README.md).

## Слайсы

**Слайс** — папка внутри слоя (кроме `app` и `shared`), группирующая весь код одного бизнес-домена.

### Правила слайсов

- Имя слайса **MUST** отражать бизнес-концепт: `post`, `comment`, `cart`, `add-to-cart`, `header`.
- Слайсы одного слоя **MUST NOT** зависеть друг от друга.
  - Исключение: `entities/*` через `@x`. См. [04-cross-imports.md](04-cross-imports.md).
- Связанные слайсы **MAY** группироваться в папку для удобства навигации, но:
  - **MUST NOT** держать общий код в папке группировки.
  - Папка-группа — чисто структурная.

✓ Группировка без общего кода:
```
entities/
└── commerce/
    ├── product/
    │   └── index.ts
    └── order/
        └── index.ts
```

✗ Группировка с общим кодом:
```
entities/
└── commerce/
    ├── shared-types.ts   ← ✗ нарушает изоляцию слайсов
    ├── product/
    └── order/
```

### Именование слайсов

- **MUST** использовать **kebab-case** для имён слайсов, сегментов и подпапок на всех слоях.
- Имена **MUST** описывать бизнес-концепт — без суффиксов слоя (`UserEntity`, `AddToCartFeature`, `HeaderWidget`).
- Имена файлов **SHOULD** тоже быть в kebab-case. Исключение: конвенции фреймворка, требующие другого регистра для конкретного типа файлов (например, React-компоненты традиционно называют `PascalCase.tsx`), перекрывают это правило **только для этого типа файлов** — это конвенция фреймворка, не FSD.

| ✓ Корректно     | ✗ Некорректно         | Почему                                |
|-----------------|-----------------------|---------------------------------------|
| `user`          | `UserEntity`          | Бизнес-существительное без суффикса   |
| `add-to-cart`   | `AddToCartFeature`    | Бизнес-действие без суффикса слоя     |
| `header`        | `HeaderWidget`        | Бизнес-роль без суффикса слоя         |
| `sign-in`       | `loginPage`           | Kebab-case; ориентировано на бизнес   |

## Сегменты

Внутри каждого слайса (и напрямую в `app` / `shared`) код делится по **техническому назначению** на **сегменты**.

### Стандартные сегменты

| Сегмент  | Содержит                                                                            |
|----------|-------------------------------------------------------------------------------------|
| `ui`     | UI-компоненты, стили, UI-форматтеры                                                 |
| `api`    | Функции запросов, DTO, мапперы ответов                                              |
| `model`  | Стораджи, схемы, бизнес-логика, доменные типы                                       |
| `lib`    | Локальные для слайса утилиты (чистые функции, адаптеры, hooks-как-утилиты)           |
| `config` | Feature flags, локальная конфигурация слайса                                        |

### Правило именования сегмента

Сегмент **MUST** описывать **назначение** (для чего этот код), а не **сущность** (чем он является).

✗ Запрещённые имена:
- `components/`
- `hooks/`
- `types/`
- `utils/`
- `helpers/`
- `constants/` (используйте `config/` или инлайн)

✓ Обязательные имена: `ui`, `api`, `model`, `lib`, `config`.

### Кастомные сегменты

Кастомные сегменты **MAY** только в слоях без слайсов (`app`, `shared`). Примеры:
- `shared/routes/`
- `shared/fonts/`
- `app/providers/`

В `pages`, `widgets`, `features`, `entities` — кастомных сегментов **MUST NOT**, придерживайтесь стандартной пятёрки.

## Пример: entity-слайс

✓ Корректно:
```
entities/user/
├── index.ts              ← Public API
├── ui/
│   ├── user-card.tsx
│   └── user-avatar.tsx
├── api/
│   ├── user-api.ts       ← функции запросов
│   └── user-dto.ts       ← серверные типы
├── model/
│   ├── store.ts
│   ├── types.ts          ← доменные типы
│   └── selectors.ts
├── lib/
│   └── format-user-name.ts
└── @x/
    └── post.ts           ← фасад кросс-импорта
```

✗ Некорректно:
```
entities/user/
├── components/           ← ✗ имя по сущности
├── hooks/                ← ✗ имя по сущности
├── types.ts              ← ✗ вне сегмента
├── UserUtils.ts          ← ✗ вне сегмента, PascalCase
└── (нет index.ts)        ← ✗ отсутствует Public API
```

## Пример: feature-слайс

✓ Корректно:
```
features/add-to-cart/
├── index.ts
├── ui/
│   └── add-to-cart-button.tsx
├── api/
│   └── add-to-cart.ts
├── model/
│   ├── use-add-to-cart.ts
│   └── types.ts
└── config/
    └── flags.ts
```

## Пример: сегменты shared

✓ Корректно:
```
shared/
├── ui/
│   ├── button/
│   │   ├── index.ts
│   │   └── button.tsx
│   └── input/
│       ├── index.ts
│       └── input.tsx
├── api/
│   ├── http-client.ts
│   └── errors.ts
├── lib/
│   ├── dates/            ← подпапка по назначению
│   │   └── format.ts
│   └── collections/
│       └── group-by.ts
└── config/
    ├── env.ts
    └── routes.ts
```

Папку `shared/lib` **SHOULD** разбивать на подпапки по назначению (`dates`, `collections`, `validation`), а не сваливать в плоскую кучу.

# AGENTS.md — Onboarding (первый день форка)

> Категорийная точка входа для ИИ-агентов и людей, только что склонировавших шаблон.
> Главы написаны на русском. Покрывают адопцию template-репо: что сделать в первый день, как ветвиться под специфику проекта.

---

## Scope

Эта категория покрывает **первые шаги после форка template-репо**:

- Чек-лист первого дня: от клонирования до первого коммита с зелёным линтером.
- Развилки под типовые сценарии: монорепо, fullstack, Expo, Next.js SSR-first, solo-репо без CI.
- Red-flag'и: признаки того, что что-то идёт не так.
- Навигация «что читать после онбординга» по остальным категориям.

**Не покрывает**:

- Содержательные правила (архитектура, код-стайл, workflow, security) — это остальные категории.
- Квартальный governance-ритуал и smoke-tests — это [`../_meta/`](../_meta/AGENTS.md).
- Продуктовую документацию (домен, API, операции) — это добавляется в форк через собственные категории.

## Working Protocol

Для первого дня после форка:

1. **Откройте [`01-first-fork.md`](01-first-fork.md)** — это основной документ.
2. **Проходите чек-лист последовательно** — шаги зависят друг от друга (нельзя сделать шаг 5 до шага 3).
3. **Если шаг не подходит** — сверьтесь с [`02-common-branches.md`](02-common-branches.md): возможно, ваш сценарий там описан.
4. **Если не нашли ответ** — откройте issue в template-репо со сценарием; это сигнал, что документация нуждается в обновлении.
5. **Перед первым коммитом** — Pre-Flight Checklist ниже.

Для второго и последующих форков этой же команды: `01-first-fork.md` пробегается быстрее (знакомая территория), но **MUST** проходиться целиком — поля `PROFILE.md` меняются между проектами.

## Hard Invariants

- [`../code-style/PROFILE.md`](../code-style/PROFILE.md) **MUST** быть заполнен в первом PR после форка. `TODO` во всех полях после форк-адопции — **MUST NOT**.
- Корневой [`../../README.md`](../../README.md) **MUST** быть переписан под конкретный проект (replace: название, описание, использование). Template-версия **MUST NOT** оставаться как есть.
- `CHANGELOG.md` **MUST** быть очищен от template-записей и начат заново для проекта (либо полностью удалён до первого релиза).
- Linter (`npm run lint`) **MUST** проходить перед первым коммитом в ветке форка.
- Отличия от template-дефолтов (другой стек, другая процесс-модель) **MUST** фиксироваться через override-ADR в `docs/decisions/` форка.
- Повторное использование `id` ADR из template'а **MUST NOT**: новый ADR в форке начинается с `id: 0011` (или следующего после последнего template'ного).

## Pre-Flight Checklist

Перед первым коммитом в форкнутом проекте:

- [ ] [`../code-style/PROFILE.md`](../code-style/PROFILE.md) заполнен реальными значениями; все поля — либо конкретное значение, либо `N/A` с обоснованием в комментарии; `[x]` проставлены напротив активных variant-глав.
- [ ] Корневой [`../../README.md`](../../README.md) описывает **ваш проект**, не шаблон.
- [ ] [`../../CHANGELOG.md`](../../CHANGELOG.md) очищен от template-записей.
- [ ] [`../../.github/CODEOWNERS`](../../.github/CODEOWNERS) заполнен (либо файл удалён, если solo-репо).
- [ ] LICENSE-файл создан (MIT / Apache-2.0 / proprietary / другой) — корневой README **MUST** отражать выбор.
- [ ] `.env.example` содержит все env-переменные проекта с описанием происхождения значений.
- [ ] `node scripts/lint-docs.mjs` — зелёный.
- [ ] Прочитаны `AGENTS.md` всех 6 content-категорий (architecture, code-style, workflow, security, decisions, _meta) — бегло, по Scope.
- [ ] Первый коммит следует формату [`../workflow/01-git.md`](../workflow/01-git.md) раздел 2; commit message — `chore: adopt documentation template` или аналогичный.
